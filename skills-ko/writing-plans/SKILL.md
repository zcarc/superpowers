# 플랜 작성 (Writing Plans)

## 개요

구현 작업을 실행 가능하고 검증 가능한 작은 단계들로 분해합니다. 이 계획은 신뢰할 수 있는 핸드오프 아티팩트로서, 여러분 자신이나 다른 하위 에이전트가 최소한의 의사소통으로 구현을 완료할 수 있도록 해야 합니다.

## 대상 독자

상대방이 유능한 개발자라고 가정하되, 그 도구 세트나 문제 영역에 대해서는 거의 모른다고 생각하십시오. 또한 좋은 테스트 설계에 대해 잘 모를 수 있다고 가정하십시오.

**시작 시 선언:** "구현 계획을 작성하기 위해 writing-plans 기술을 사용하고 있습니다."

**컨텍스트:** Full Lane 계획은 전용 작업트리(worktree)에서 실행되어야 합니다. Light Lane 작업은 적절한 경우 현재 작업 공간에 머무를 수 있습니다.

**계획 저장 위치:** `docs/superpowers/plans/YYYY-MM-DD-<feature-name>.md`
- (계획 위치에 대한 사용자 선호도가 있는 경우 이 기본값보다 우선합니다)

## 소규모 범위 예외

승인된 설계가 명확한 파일과 수락 기준을 가진 작고 국소적인 변경인 경우:
- 전체 계획 문서를 작성하지 않아도 됩니다.
- 구현을 위한 짧고 가벼운 추적 가능한 작업 목록을 생성하십시오.
- 대화 중에 동일한 작업 목록을 인라인으로 제공하십시오.

다음과 같은 경우 저장된 계획 문서를 사용하십시오:
- 여러 서브시스템이 관련된 경우
- 작업 순서가 중요한 경우
- 핸드오프의 가치가 의미 있는 경우
- 사용자가 명시적으로 계획 문서를 요청한 경우

## 레인 인식 계획 (Lane-Aware Planning)

Light Lane이 이미 선택된 경우:
- 기본적으로 계획을 인라인으로 유지합니다.
- 작업이 이미 에스컬레이션되지 않은 한 소규모 범위 예외(Small-Scope Exception)를 사용합니다.
- 세 가지 모드 실행 선택기를 제시하지 마십시오.
- 에스컬레이션할 구체적인 이유가 없는 한 인라인 실행(Inline Execution)을 암시적으로 권장합니다.

Full Lane이 이미 선택된 경우:
- 일반적인 저장된 계획 흐름을 사용합니다.
- 선택이 비용이나 안전에 의미 있게 영향을 미칠 때만 실행 모드 선택을 제시합니다.

## 범위 확인

사양이 여러 독립적인 서브시스템을 다루는 경우, 브레인스토밍 단계에서 서브 프로젝트 사양으로 분리되었어야 합니다. 그렇지 않았다면 서브시스템당 하나씩 별도의 계획으로 나눌 것을 제안하십시오. 각 계획은 그 자체로 작동하고 테스트 가능한 소프트웨어를 생성해야 합니다.

## 파일 구조

작업을 정의하기 전에 어떤 파일이 생성되거나 수정될지, 그리고 각 파일의 책임이 무엇인지 매핑하십시오. 이곳이 분해 결정이 확정되는 지점입니다.

- 명확한 경계와 잘 정의된 인터페이스를 가진 디자인 유닛을 설계하십시오. 각 파일은 하나의 명확한 책임을 가져야 합니다.
- 한 번에 컨텍스트로 파악할 수 있는 코드에 대해 가장 잘 추론할 수 있으며, 파일이 집중되어 있을 때 수정이 더 신뢰할 수 있습니다. 너무 많은 일을 하는 큰 파일보다 작고 집중된 파일을 선호하십시오.
- 함께 변경되는 파일은 함께 있어야 합니다. 기술 레이어가 아닌 책임에 따라 나누십시오.
- 기존 코드베이스에서는 확립된 패턴을 따르십시오. 코드베이스가 큰 파일을 사용한다면 일방적으로 구조를 변경하지 마십시오. 하지만 수정 중인 파일이 너무 커졌다면 계획에 분할 작업을 포함하는 것이 합리적입니다.

이 구조는 작업 분해의 기초가 됩니다. 각 작업은 그 자체로 독립적으로 의미가 있는 자기 완비적인(self-contained) 변경 사항을 생성해야 합니다.

## 한 입 크기의 작업 세밀도

**각 단계는 하나의 동작(2~5분)이어야 합니다:**
- "실패하는 테스트 작성" - 단계
- "실패하는지 확인하기 위해 실행" - 단계
- "테스트를 통과시키기 위한 최소한의 코드 구현" - 단계
- "테스트를 실행하고 통과하는지 확인" - 단계
- "커밋" - 단계

## 계획 문서 헤더

**모든 계획은 반드시 다음 헤더로 시작해야 합니다:**

```markdown
# [기능 이름] 구현 계획

> **에이전트 작업자 주의사항:** 필수 하위 기술: 이 계획에 가장 적합한 실행 모드를 선택하십시오: superpowers:executing-plans, superpowers:subagent-driven-development, 또는 superpowers:parallel-subagent-execution. 단계는 추적을 위해 체크박스 (`- [ ]`) 구문을 사용합니다.

**목표:** [이 작업을 통해 무엇을 만드는지 한 문장으로 설명]

**아키텍처:** [접근 방식에 대한 2~3문장]

**기술 스택:** [주요 기술/라이브러리]

---
```

## 작업 구조

````markdown
### 작업 N: [컴포넌트 이름]

**파일:**
- 생성: `exact/path/to/file.py`
- 수정: `exact/path/to/existing.py:123-145`
- 테스트: `tests/exact/path/to/test.py`

- [ ] **1단계: 실패하는 테스트 작성**

```python
def test_specific_behavior():
    result = function(input)
    assert result == expected
```

- [ ] **2단계: 테스트를 실행하여 실패 확인**

실행: `pytest tests/path/test.py::test_name -v`
예상 결과: "function not defined"와 함께 FAIL

- [ ] **3단계: 최소 구현 작성**

```python
def function(input):
    return expected
```

- [ ] **4단계: 테스트를 실행하여 통과 확인**

실행: `pytest tests/path/test.py::test_name -v`
예상 결과: PASS

- [ ] **5단계: 커밋**

```bash
git add tests/path/test.py src/path/file.py
git commit -m "feat: add specific feature"
```
````

## 플레이스홀더 금지

모든 단계에는 엔지니어가 필요로 하는 실제 내용이 포함되어야 합니다. 다음은 **계획 실패**의 예시이며, 절대 작성하지 마십시오:
- "TBD", "TODO", "나중에 구현", "세부 사항 채우기"
- "적절한 에러 핸들링 추가" / "유효성 검사 추가" / "예외 케이스 처리"
- "위 작업에 대한 테스트 작성" (실제 테스트 코드 없이)
- "작업 N과 유사함" (코드를 반복해서 작성하십시오 - 엔지니어는 순서 없이 작업을 읽을 수 있습니다)
- 어떻게(how) 수행하는지 보여주지 않고 무엇을 할지만 설명하는 단계 (코드 단계에는 코드 블록이 필수)
- 어떤 작업에서도 정의되지 않은 타입, 함수 또는 메서드에 대한 참조

## 기억할 것
- 항상 정확한 파일 경로 사용
- 모든 단계에 완전한 코드 포함 - 단계가 코드를 변경한다면 코드를 보여주십시오
- 예상 출력과 함께 정확한 명령 제공
- DRY, YAGNI, TDD, 빈번한 커밋

## 자체 검토

전체 계획을 작성한 후, 새로운 시각으로 사양을 다시 보고 계획과 대조해 보십시오. 이는 여러분이 직접 수행하는 체크리스트이며, 하위 에이전트를 파견하는 것이 아닙니다.

**1. 사양 커버리지:** 사양의 각 섹션/요구사항을 훑어보십시오. 이를 구현하는 작업을 지목할 수 있습니까? 누락된 부분을 리스트업하십시오.

**2. 플레이스홀더 스캔:** 계획에서 위에 언급된 "플레이스홀더 금지" 섹션의 레드 플래그 패턴을 찾으십시오. 발견되면 수정하십시오.

**3. 타입 일관성:** 나중 작업에서 사용한 타입, 메서드 시그니처, 속성 이름이 이전 작업에서 정의한 것과 일치합니까? 작업 3에서는 `clearLayers()`라고 불리는 함수가 작업 7에서 `clearFullLayers()`로 불린다면 이는 버그입니다.

문제 발견 시 인라인으로 수정하십시오. 재검토할 필요 없이 수정하고 다음으로 넘어가십시오. 작업을 찾지 못한 사양 요구사항이 있다면 작업을 추가하십시오.

## 리뷰 용어 (Review Terms)

- **구현자 자체 검토 (Implementer self-review)** - 작업 담당자가 핸드오프 전에 자신의 작업을 검토합니다. 이것은 로컬 품질 검사이며 공식 코드 리뷰가 아닙니다.
- **통합 검증 (Integration verification)** - 작업이나 웨이브가 통합된 후 전체 결과물이 여전히 작동하는지 확인하기 위해 실행하는 테스트 또는 검사입니다. 이것은 공식 코드 리뷰가 아닙니다.
- **공식 리뷰 (Formal review)** - 완료된 전체 통합 결과물에 대해 명시적으로 `superpowers:requesting-code-review`를 사용하는 것을 말합니다. 기본 시점: Full Lane 작업에는 필수이며, 리뷰가 명시적으로 요청되거나 임계값이 트리거되지 않는 한 Light Lane 작업에는 선택 사항입니다.
- `superpowers:receiving-code-review`는 공식 리뷰에서 검토가 필요한 피드백을 반환한 경우에만 적용됩니다.

## 실행 핸드오프 (Execution Handoff)

Light Lane이 이미 선택된 경우 세 가지 모드 실행 선택기를 제시하지 마십시오. 작업이 이미 에스컬레이션되지 않은 한 인라인 작업 목록 또는 저장된 계획을 인라인 실행 아티팩트로 간주하십시오.

Full Lane이 이미 선택되었고 전체 계획 문서를 저장했다면, 해당 계획을 분석하고 사용자에게 선택을 요청하기 전에 가장 적합한 실행 모드를 권장하십시오.

Full Lane이 이미 선택되었고 소규모 범위 추적 작업 목록을 생성했다면, 해당 작업 목록을 계획 아티팩트로 간주하고 별도로 가장 적합한 실행 모드를 권장하십시오.

평가 요소:
- 작업 의존성
- 작업 간 파일 중복
- 통합 리스크
- 인터페이스의 모호함
- 병렬 실행으로 예상되는 속도 향상
- 컨트롤러의 세밀한 감시 필요성

그런 다음 이 형식으로 권장안을 제시하십시오:

**권장 실행 모드:** [정확한 모드 이름]

**이 계획에 가장 적합한 이유:**
- [이유 1]
- [이유 2]
- [이유 3]

**다른 사용 가능한 모드가 가장 적합하지 않은 이유:**
- 권장 모드를 제외하고 아래에 나열된 각 실행 모드에 대해 하나의 불렛을 추가합니다.
- 선택되지 않은 각 모드를 정확히 한 번씩 다룹니다.
- 형식: `- **[정확한 모드 이름]** - [선택되지 않은 이유]`

**실행 모드:**

- **인라인 실행 (Inline Execution)**
  - 이 세션에서 계획을 작업별로 따라가며 직접 실행하고, 레인을 인식하는 리뷰 규칙에 따라 완료
  - 소규모 계획, 긴밀하게 결합된 작업, 또는 컨트롤러가 직접 구현하는 것이 좋은 상황에 적합
  - 필수 실행 기술: `superpowers:executing-plans`

- **하위 에이전트 기반 (Subagent-Driven)**
  - 작업당 하나의 신선한 하위 에이전트를 파견하여 순차적으로 실행하고, 각 구현자가 자체 검토를 하도록 한 뒤, 모든 작업 완료 후 최종 공식 리뷰를 한 번 수행
  - 작업 격리가 도움이 되지만 병렬 실행이 리스크를 가중시킬 수 있는 Full Lane 다중 작업 계획에 가장 적합한 기본 옵션
  - 필수 실행 기술: `superpowers:subagent-driven-development`

- **병렬 하위 에이전트 (Parallel Subagents)**
  - 계획을 독립적이고 충돌이 없는 웨이브(waves)로 분할하여 웨이브 내 작업마다 하나의 신선한 하위 에이전트를 실행하고, 각 구현자가 자체 검토를 하도록 하며, 통합된 각 웨이브를 검증하고 나서, 모든 웨이브가 통합된 후 최종 공식 리뷰를 한 번 수행
  - 소유권이 명확하고 결합도가 낮으며 병렬화 시 속도 향상이 뚜렷한 Full Lane 계획에 적합
  - 필수 실행 기술: `superpowers:parallel-subagent-execution`

**어떤 실행 모드를 사용하시겠습니까?**

사용자가 모드를 선택한 후:
- 해당 모드에 필요한 실행 기술(execution skill)을 로드합니다.
- 해당 모드의 워크플로우를 정확히 따릅니다.

모드별 주요 사항:

- **인라인 실행 (Inline Execution)**
  - 이 세션에서 계획을 작업별로 따라가며 직접 실행
  - 리뷰 동작은 레인에 따라 다름: Light Lane은 집중적인 검증과 로컬 자체 검사로 완료할 수 있는 반면, Full Lane은 모든 작업이 완료된 후 최종 공식 리뷰를 실행함

- **하위 에이전트 기반 (Subagent-Driven)**
  - 작업당 신선한 하위 에이전트 + 구현자 자체 리뷰
  - 모든 작업이 완료된 후 공식 리뷰 1회 수행

- **병렬 하위 에이전트 (Parallel Subagents)**
  - 계획을 종속성 안전 웨이브로 분할
  - 작업당 명시적인 파일 소유권 할당
  - 계속 진행하기 전에 통합된 각 웨이브 검증
  - 모든 웨이브가 통합된 후 공식 리뷰 1회 수행
