# 계획 작성하기

## 개요

엔지니어가 우리 코드베이스에 대한 문맥이 전혀 없고 취향도 미심쩍다는 가정하에 포괄적인 구현 계획을 작성하십시오. 각 작업에 대해 수정해야 할 파일, 코드, 테스트, 확인해야 할 문서, 테스트 방법 등 엔지니어가 알아야 할 모든 것을 문서화하십시오. 전체 계획을 한입 크기(bite-sized)의 작업들로 제공하십시오. DRY, YAGNI, TDD 원칙을 지키고 빈번하게 커밋하십시오.

엔지니어가 숙련된 개발자지만, 우리의 도구 모음이나 문제 영역에 대해서는 거의 모른다고 가정하십시오. 또한 좋은 테스트 설계에 대해 잘 모를 수 있다고 가정하십시오.

**시작 시 공지:** "구현 계획을 생성하기 위해 writing-plans 기술을 사용하고 있습니다."

**문맥:** 이 작업은 전용 작업 테리토리(`worktree` - brainstorming 기술에 의해 생성됨)에서 실행되어야 합니다.

**계획 저장 위치:** `docs/superpowers/plans/YYYY-MM-DD-<feature-name>.md`
- (계획 위치에 대한 사용자 설정이 있는 경우 이 기본값보다 우선합니다)

## 범위 확인 (Scope Check)

사양이 여러 개의 독립적인 하위 시스템을 다루는 경우, 브레인스토밍 단계에서 하위 프로젝트 사양으로 분리되었어야 합니다. 그렇지 않은 경우, 하위 시스템당 하나씩 별도의 계획으로 나눌 것을 제안하십시오. 각 계획은 그 자체로 작동하고 테스트 가능한 소프트웨어를 생산해야 합니다.

## 파일 구조

작업을 정의하기 전에 어떤 파일이 생성되거나 수정될지, 그리고 각 파일이 어떤 책임을 갖는지 매핑하십시오. 이 단계에서 분해 결정이 확정됩니다.

- 명확한 경계와 잘 정의된 인터페이스를 가진 설계 단위를 만드십시오. 각 파일은 하나의 명확한 책임을 가져야 합니다.
- 한 번에 문맥을 파악할 수 있는 코드가 가장 추론하기 좋으며, 파일이 집중되어 있을 때 수정이 더 안정적입니다. 너무 많은 일을 하는 큰 파일보다 작은 전문화된 파일을 선호하십시오.
- 함께 변경되는 파일은 함께 있어야 합니다. 기술 계층이 아닌 책임에 따라 나누십시오.
- 기존 코드베이스에서는 확립된 패턴을 따르십시오. 코드베이스가 큰 파일을 사용하는 경우 일방적으로 재구성하지 마십시오. 하지만 수정 중인 파일이 너무 감당하기 힘들게 커졌다면 계획에 분할 과정을 포함하는 것이 타당합니다.

이 구조는 작업 분해의 기준이 됩니다. 각 작업은 그 자체로 의미가 있는 독립적인 변경 사항을 만들어내야 합니다.

## 한입 크기 작업의 세분도 (Bite-Sized Task Granularity)

**각 단계는 하나의 동작(2~5분)이어야 합니다:**
- "실패하는 테스트 작성" - 단계
- "테스트를 실행하여 실패하는지 확인" - 단계
- "테스트를 통과시키기 위한 최소한의 코드 구현" - 단계
- "테스트를 실행하여 통과하는지 확인" - 단계
- "커밋" - 단계

## 계획 문서 헤더

**모든 계획은 반드시 이 헤더로 시작해야 합니다:**

```markdown
# [기능 이름] 구현 계획

> **에이전트 작업자용:** 필수 하위 기술: 이 계획을 작업별로 구현하려면 superpowers:subagent-driven-development (권장) 또는 superpowers:executing-plans를 사용하십시오. 진행 상황 추적을 위해 체크박스(`- [ ]`) 구문을 사용합니다.

**목표:** [이것이 무엇을 만드는지 설명하는 한 문장]

**아키텍처:** [접근 방식에 대한 2~3문장]

**기술 스택:** [주요 기술/라이브러리]

---
```

## 작업 구조

````markdown
### 작업 N: [컴포넌트 이름]

**파일:**
- 생성: `exact/path/to/file.py`
- 수정: `exact/path/to/existing.py:123-145`
- 테스트: `tests/exact/path/to/test.py`

- [ ] **단계 1: 실패하는 테스트 작성**

```python
def test_specific_behavior():
    result = function(input)
    assert result == expected
```

- [ ] **단계 2: 테스트를 실행하여 실패 확인**

실행: `pytest tests/path/test.py::test_name -v`
예상 결과: "function not defined"와 함께 실패(FAIL)

- [ ] **단계 3: 최소 구현 작성**

```python
def function(input):
    return expected
```

- [ ] **단계 4: 테스트를 실행하여 통과 확인**

실행: `pytest tests/path/test.py::test_name -v`
예상 결과: 통과(PASS)

- [ ] **단계 5: 커밋**

```bash
git add tests/path/test.py src/path/file.py
git commit -m "feat: 특정 기능 추가"
```
````

## 자리 표시자(Placeholder) 금지

모든 단계에는 엔지니어가 필요한 실제 내용이 포함되어야 합니다. 다음은 **계획 실패**의 예이며, 절대 작성하지 마십시오:
- "TBD", "TODO", "나중에 구현", "세부 사항 채우기"
- "적절한 에러 핸들링 추가" / "검증 추가" / "예외 케이스 처리"
- "위의 사항에 대한 테스트 작성" (실제 테스트 코드 없이)
- "작업 N과 유사함" (코드를 반복하십시오 — 엔지니어는 작업을 순서대로 읽지 않을 수 있습니다)
- 어떻게 하는지 보여주지 않고 무엇을 할지만 설명하는 단계 (코드 단계에는 코드 블록이 필수임)
- 어떤 작업에서도 정의되지 않은 타입, 함수 또는 메서드 참조

## 기억할 것
- 항상 정확한 파일 경로 기재
- 모든 단계에 전체 코드 포함 — 단계가 코드를 변경한다면 그 코드를 보여주십시오
- 예상 결과가 포함된 정확한 명령문
- DRY, YAGNI, TDD, 빈번한 커밋

## 자체 검토

전체 계획을 작성한 후, 새로운 시선으로 사양을 다시 살펴보고 계획과 대조해 보십시오. 이것은 여러분이 직접 수행하는 체크리스트이며 서브에이전트 파견이 아닙니다.

**1. 사양 충족 여부:** 사양의 각 섹션/요구사항을 훑어보십시오. 그것을 구현하는 구체적인 작업이 있습니까? 누락된 부분을 나열하십시오.

**2. 자리 표시자 스캔:** 계획에서 위에 언급된 "자리 표시자 금지" 패턴과 같은 주의 신호를 검색하십시오. 수정하십시오.

**3. 타입 일관성:** 이후 작업에서 사용한 타입, 메서드 서명 및 속성 이름이 이전 작업에서 정의한 것과 일치합니까? 작업 3에서는 `clearLayers()`라고 불렀는데 작업 7에서 `clearFullLayers()`라고 부르는 것은 버그입니다.

문제를 발견하면 즉시 수정하십시오. 다시 검토할 필요 없이 수정하고 다음으로 넘어가면 됩니다. 사양 요구사항 중 작업이 없는 것을 발견하면 해당 작업을 추가하십시오.

## 실행 핸드오버 (Execution Handoff)

계획을 저장한 후, 실행 방식을 제안하십시오:

**"계획이 완료되어 `docs/superpowers/plans/<파일명>.md`에 저장되었습니다. 두 가지 실행 옵션이 있습니다:**

**1. 서브에이전트 주도 (권장)** - 작업당 신선한 서브에이전트를 파견하며, 구현 문맥을 격리하고 모든 작업이 완료된 후 한 번의 최종 리뷰를 수행합니다.

**2. 인라인 실행** - executing-plans를 사용하여 이 세션에서 작업을 실행하며, 계획을 한 단계씩 따르며 직접 실행합니다.

**어떤 방식을 선택하시겠습니까?"**

**서브에이전트 주도 선택 시:**
- **필수 하위 기술:** superpowers:subagent-driven-development 사용
- 작업당 신선한 서브에이전트 + 작업당 구현자 셀프 리뷰 + 모든 작업 후 한 번의 최종 코드 리뷰

**인라인 실행 선택 시:**
- **필수 하위 기술:** superpowers:executing-plans 사용
- 이 세션에서 계획을 작업별로 따라가며 직접 실행
