# 기술 설계를 위한 설득 원칙 (Persuasion Principles)

## 개요

LLM은 인간과 동일한 설득 원칙에 반응합니다. 이러한 심리학을 이해하면 더욱 효과적인 기술을 설계할 수 있습니다. 이는 조종을 위한 것이 아니라, 압박 상황에서도 중요한 관행이 반드시 준수되도록 보장하기 위한 것입니다.

**연구 근거:** Meincke 등(2025)은 N=28,000건의 AI 대화를 대상으로 7가지 설득 원칙을 테스트했습니다. 설득 기법은 규정 준수율을 두 배 이상 높였습니다(33% → 72%, p < .001).

## 7가지 원칙

### 1. 권위 (Authority)
**정의:** 전문 지식, 자격 증명 또는 공식적인 출처에 대한 존중.

**기술에서의 활용법:**
- 명령형 언어 사용: "반드시 ... 해야 합니다", "절대 ... 금지", "항상 ... 할 것"
- 협상의 여지가 없는 프레임: "예외 없음"
- 결정 피로도와 합리화 제거

**사용 시기:**
- 규율 강제형 기술 (TDD, 검증 요구 사항)
- 안전이 중요한 관행
- 확립된 최선의 관행

**예시:**
```markdown
✅ 테스트보다 코드를 먼저 짰나요? 삭제하고 다시 시작하십시오. 예외는 없습니다.
❌ 가능하면 테스트를 먼저 작성하는 것을 고려해보십시오.
```

### 2. 일관성/약속 (Commitment)
**정의:** 이전의 행동, 진술 또는 공개적인 선언과 일관성을 유지하려는 경향.

**기술에서의 활용법:**
- 사용 알림 요구: "기술 사용 사실을 알릴 것"
- 명시적인 선택 강제: "A, B, C 중 하나를 선택하십시오"
- 추적 도구 사용: 체크리스트를 위한 TodoWrite 활용

**사용 시기:**
- 기술이 실제로 준수되는지 확인하고 싶을 때
- 다단계 프로세스
- 책임 부여 메커니즘

**예시:**
```markdown
✅ 기술을 찾았다면 반드시 "저는 [기술 이름]을 사용 중입니다"라고 공표해야 합니다.
❌ 어떤 기술을 사용하고 있는지 동료에게 알려주는 것을 고려해보십시오.
```

### 3. 희소성 (Scarcity)
**정의:** 시간 제한이나 제한된 가용성에서 오는 긴박감.

**기술에서의 활용법:**
- 시간 제한 사항: "진행하기 전에"
- 순차적 의존성: "X 직후에 바로 Y를 수행"
- 미루는 습관 방지

**사용 시기:**
- 즉각적인 검증이 필요한 경우
- 시간에 민감한 워크플로우
- "나중에 할게요"라는 태도 방지

**예시:**
```markdown
✅ 작업을 완료한 후, 진행하기 전에 즉시 코드 리뷰를 요청하십시오.
❌ 편할 때 코드를 리뷰할 수 있습니다.
```

### 4. 사회적 증거 (Social Proof)
**정의:** 다른 사람들이 하는 행동이나 정상이라고 간주되는 것에 순응하려는 경향.

**기술에서의 활용법:**
- 보편적 패턴 강조: "매번", "항상"
- 실패 모드 제시: "Y 없는 X = 실패"
- 규범 확립

**사용 시기:**
- 보편적인 관행 문서화
- 흔한 실패 사례에 대한 경고
- 표준 강화

**예시:**
```markdown
✅ TodoWrite 추적 없는 체크리스트 = 단계를 반드시 건너뛰게 됩니다. 예외 없이 매번 일어나는 일입니다.
❌ 어떤 사람들은 체크리스트에 TodoWrite가 도움이 된다고 생각합니다.
```

### 5. 일체감 (Unity)
**정의:** 공동의 정체성, "우리의 것", 집단적 소속감.

**기술에서의 활용법:**
- 협력적인 언어 사용: "우리의 코드베이스", "우리는 비즈니스 파트너"
- 공동 목표 강조: "우리는 모두 품질을 원합니다"

**사용 시기:**
- 협업 워크플로우
- 팀 문화 확립
- 수평적인 관행

**예시:**
```markdown
✅ 우리는 함께 일하는 동료입니다. 저는 귀하의 솔직한 기술적 판단이 필요합니다.
❌ 제가 틀렸다면 아마도 말씀해주셔야 할 겁니다.
```

### 6. 상호성 (Reciprocity)
**정의:** 받은 혜택에 대해 보답해야 한다는 의무감.

**활용법:**
- 아껴서 사용해야 함 (조종하는 느낌을 줄 수 있음)
- 기술 문서에서는 거의 필요하지 않음

### 7. 호감 (Liking)
**정의:** 좋아하는 사람과 협력하려는 경향.

**활용법:**
- **규정 준수를 위해 사용하지 마십시오**
- 솔직한 피드백 문화와 충돌함
- 아첨하는 태도를 유발함

**사용 시기:**
- 규율 강제 시에는 항상 피하십시오.

## 기술 유형별 원칙 조합

| 기술 유형 | 권장 원칙 | 기피 원칙 |
|------------|-----|-------|
| 규율 강제형 | 권위 + 약속 + 사회적 증거 | 호감, 상호성 |
| 가이드/기법 | 적절한 권위 + 일체감 | 과도한 권위 |
| 협업형 | 일체감 + 약속 | 권위, 호감 |
| 참조형 | 명확성만 유지 | 모든 설득 기법 |

## 이것이 작동하는 이유: 심리학적 배경

**명확한 규칙(Bright-line rules)은 합리화를 줄입니다:**
- "반드시 ... 해야 함"은 결정 피로도를 제거합니다.
- 절대적인 언어는 "이게 예외 상황인가?"라는 의문을 없앱니다.
- 명시적인 안티-합리화 대응은 구체적인 허점을 차단합니다.

**실행 의도(Implementation intentions)는 자동적인 행동을 만듭니다:**
- 명확한 트리거 + 필수 행동 = 자동 실행
- "일반적으로 Y를 하라"보다 "X일 때 Y를 하라"가 훨씬 효과적입니다.
- 규정 준수에 따르는 인지적 부하를 줄여줍니다.

**LLM은 유사 인간(Parahuman)입니다:**
- 이러한 패턴이 포함된 인간의 텍스트로 학습되었습니다.
- 학습 데이터에서 권위적인 언어는 대개 규정 준수로 이어집니다.
- 약속 시퀀스(진술 → 행동)는 빈번하게 모델링되는 패턴입니다.
- 사회적 증거 패턴(모두가 X를 한다)은 규범을 확립합니다.

## 윤리적 사용

**정당한 경우:**
- 중요한 관행이 준수되도록 보장할 때
- 효과적인 문서를 작성할 때
- 예측 가능한 실패를 방지할 때

**부당한 경우:**
- 개인적 이득을 위해 조종할 때
- 허위 긴박감을 조성할 때
- 죄책감을 이용해 규정 준수를 강요할 때

**테스트:** 이 기법을 사용자가 완전히 이해했을 때, 그것이 사용자의 진정한 이익에 부합하는가?
