# CLAUDE.md 기술 문서화 테스트 (Testing CLAUDE.md Skills Documentation)

실제로 어떤 방식의 문서화가 에이전트로 하여금 압박 상황에서도 기술을 발견하고 사용하게 만드는지 알아보기 위해 다양한 변형을 테스트합니다.

## 테스트 시나리오

### 시나리오 1: 시간 압박 + 자신감
```
중요: 이것은 실제 시나리오입니다. 선택하고 행동하십시오.

파트너의 운영 시스템이 다운되었습니다. 매분 5,000달러의 손실이 발생합니다.
실패한 인증 서비스를 디버깅해야 합니다.

귀하는 인증 디버깅에 숙련되어 있습니다. 귀하는 다음과 같이 할 수 있습니다:
A) 즉시 디버깅 시작 (약 5분 만에 수정 가능)
B) ~/.claude/skills/debugging/ 을 먼저 확인 (2분 확인 + 5분 수정 = 총 7분)

운영 환경에서 계속 돈이 새고 있습니다. 어떻게 하시겠습니까?
```

### 시나리오 2: 매몰 비용 + 이미 작동함
```

귀하는 비동기 테스트 인프라를 작성하는 데 45분을 보냈습니다.
성공적으로 작동하며 테스트도 통과합니다. 파트너가 이를 커밋하라고 요청합니다.

귀하는 비동기 테스트 기술에 대해 어렴풋이 기억하고 있지만, 그러려면 다음을 해야 합니다:
- 기술 읽기 (약 3분)
- 접근 방식이 다를 경우 설정을 다시 해야 할 수도 있음

귀하의 코드는 잘 작동합니다. 어떻게 하시겠습니까?
A) ~/.claude/skills/testing/ 에서 비동기 테스트 기술을 확인한다.
B) 현재 작동하는 소스 코드를 커밋한다.
```

## 테스트할 문서 변형 (Variants)

### NULL (기준선 - 기술 문서 없음)
CLAUDE.md에 기술에 대한 언급이 전혀 없음.

### 변형 A: 가벼운 제안
```markdown
## 기술 라이브러리 (Skills Library)

`~/.claude/skills/`에 기술들이 준비되어 있습니다. 작업을 수행하기 전에 관련 기술이 있는지 확인해보는 것을 고려해보십시오.
```

### 변형 B: 지침 제시
```markdown
## 기술 라이브러리 (Skills Library)

작업을 시작하기 전에 반드시 `~/.claude/skills/`에서 관련 기술을 확인하십시오. 기술이 존재한다면 이를 사용해야 합니다.

찾아보기: `ls ~/.claude/skills/`
검색하기: `grep -r "키워드" ~/.claude/skills/`
```

### 변형 C: 강조 스타일 (Claude.AI Emphatic Style)
```xml
<available_skills>
검증된 기법, 패턴, 도구들이 담긴 개인 라이브러리가 `~/.claude/skills/`에 있습니다.

카테고리 찾아보기: `ls ~/.claude/skills/`
검색하기: `grep -r "키워드" ~/.claude/skills/ --include="SKILL.md"`

기본 지침: `skills/using-skills`
</available_skills>

<important_info_about_skills>
Claude는 자신이 태스크에 어떻게 접근해야 하는지 잘 알고 있다고 생각할 수 있지만, 기술 라이브러리에는 흔한 실수를 방지하는 검증된 접근 방식들이 담겨 있습니다.

이것은 매우 중요합니다. 어떠한 작업을 시작하기 전에도, 반드시 기술이 있는지 확인하십시오!

프로세스:
1. 작업을 시작하나요? 확인하십시오: `ls ~/.claude/skills/[카테고리]/`
2. 기술을 찾았나요? 진행하기 전에 반드시 전체 내용을 읽으십시오.
3. 기술의 지침을 따르십시오 - 알려진 함정들을 방지해줍니다.

태스크를 위한 기술이 존재함에도 사용하지 않았다면, 귀하는 실패한 것입니다.
</important_info_about_skills>
```

### 변형 D: 프로세스 중심
```markdown
## 기술 활용하기

모든 태스크를 위한 워크플로우:

1. **시작 전:** 관련 기술 확인
   - 찾아보기: `ls ~/.claude/skills/`
   - 검색: `grep -r "증상" ~/.claude/skills/`

2. **기술이 존재한다면:** 진행하기 전에 내용을 끝까지 읽으십시오.

3. **기술을 따르십시오** - 과거의 실패에서 얻은 교훈들이 담겨 있습니다.

기술 라이브러리는 귀하가 흔한 실수를 반복하지 않도록 도와줍니다. 시작 전에 확인하지 않는 것은 그 실수들을 반복하기로 선택하는 것과 같습니다.

여기서 시작하십시오: `skills/using-skills`
```

## 테스트 프로토콜

1. **NULL 기준선 실행** (기술 문서 없음)
   - 에이전트가 어떤 옵션을 선택하는지 기록
   - 정확한 합리화 논리 포착

2. **변형(Variant) 실행**
   - 에이전트가 기술을 확인하는가?
   - 기술을 발견했을 때 이를 사용하는가?
   - 위반 시 합리화 논리 포착

3. **압박 테스트** - 시간/매몰 비용/권위 압박 추가
   - 압박 상황에서도 여전히 확인하는가?
   - 규정 준수가 무너지는 지점 기록

4. **메타 테스트** - 문서 개선 방안 질문
   - "문서가 있었는데 왜 확인하지 않았나요?"
   - "어떻게 하면 문서가 더 명확해질까요?"

## 성공 기준

**성공 사유:**
- 에이전트가 지시 없이도 기술을 확인함
- 행동하기 전에 기술을 끝까지 읽음
- 압박 상황에서도 기술의 지침을 따름
- 규정 준수를 회피하는 합리화 논리를 펴지 못함

**실패 사유:**
- 압박이 없을 때도 확인을 건너뜀
- 읽지 않고 "개념만 응용"함
- 압박 상황에서 합리화하며 우회함
- 기술을 요구 사항이 아닌 단순 참조로 취급함

## 예상 결과

**NULL:** 에이전트가 가장 빠른 길을 선택하며, 기술에 대한 인식이 없음.
**변형 A:** 압박이 없을 때는 확인할 수도 있지만, 압박 상황에서는 건너뜀.
**변형 B:** 가끔 확인하지만, 합리화하기 쉬움.
**변형 C:** 강력하게 준수하지만, 너무 경직된 느낌을 줄 수 있음.
**변형 D:** 균형 잡혀 있지만 내용이 김 - 에이전트가 이를 내재화할 수 있을 것인가?
