# 기술(Skill) 작성 모범 사례

> Claude가 기술을 성공적으로 발견하고 사용할 수 있도록 효과적인 기술을 작성하는 방법을 배웁니다.

훌륭한 기술은 간결하고 잘 구조화되어 있으며, 실제 사용 사례를 통해 검증된 것입니다. 이 가이드는 Claude가 기술을 효과적으로 발견하고 사용할 수 있도록 돕는 실질적인 작성 지침을 제공합니다.

기술이 작동하는 방식에 대한 개념적 배경은 [Skills overview](https://platform.claude.com/docs/en/agents-and-tools/agent-skills/overview)를 참조하십시오.

## 핵심 원칙

### 간결함이 핵심입니다

[컨텍스트 윈도우](https://platform.claude.com/docs/en/build-with-claude/context-windows)는 공공재와 같습니다. 여러분의 기술은 다음을 포함하여 Claude가 알아야 할 다른 모든 정보와 컨텍스트 윈도우를 공유합니다:

* 시스템 프롬프트
* 대화 기록
* 다른 기술들의 메타데이터
* 여러분의 실제 요청 사항

기술의 모든 토큰이 즉각적인 비용을 발생시키는 것은 아닙니다. 시작 시에는 모든 기술의 메타데이터(이름 및 설명)만 미리 로드됩니다. Claude는 기술이 관련이 있을 때만 `SKILL.md`를 읽고, 추가 파일은 필요할 때만 읽습니다. 하지만 `SKILL.md`를 간결하게 유지하는 것은 여전히 중요합니다. 일단 Claude가 이를 로드하면, 모든 토큰이 대화 기록 및 다른 컨텍스트와 경쟁하게 되기 때문입니다.

**기본 가정:** Claude는 이미 충분히 똑똑합니다.

Claude가 아직 가지고 있지 않은 컨텍스트만 추가하십시오. 각 정보에 대해 다음과 같이 자문해 보십시오:

* "Claude에게 정말 이 설명이 필요한가?"
* "Claude가 이미 이 내용을 알고 있다고 가정할 수 있는가?"
* "이 단락이 토큰 비용을 정당화할 만큼 가치가 있는가?"

**좋은 예: 간결함** (약 50 토큰):

````markdown
## PDF 텍스트 추출

텍스트 추출을 위해 pdfplumber를 사용하십시오:

```python
import pdfplumber

with pdfplumber.open("file.pdf") as pdf:
    text = pdf.pages[0].extract_text()
```
````

**나쁜 예: 너무 장황함** (약 150 토큰):

```markdown
## PDF 텍스트 추출

PDF(Portable Document Format) 파일은 텍스트, 이미지 및 기타 콘텐츠를 포함하는 일반적인 파일 형식입니다. PDF에서 텍스트를 추출하려면 라이브러리를 사용해야 합니다. PDF 처리에 사용할 수 있는 라이브러리는 많지만, 사용하기 쉽고 대부분의 경우를 잘 처리하므로 pdfplumber를 추천합니다. 먼저 pip를 사용하여 설치해야 합니다. 그런 다음 아래 코드를 사용할 수 있습니다...
```

간결한 버전은 Claude가 PDF가 무엇인지, 라이브러리가 어떻게 작동하는지 이미 알고 있다고 가정합니다.

### 적절한 자유도 설정

태스크의 취약성(fragility)과 가변성(variability)에 맞게 구체화 수준을 조절하십시오.

**높은 자유도** (텍스트 기반 지침):

다음의 경우에 사용하십시오:
* 여러 접근 방식이 모두 유효할 때
* 결정이 컨텍스트에 따라 달라질 때
* 휴리스틱(Heuristics)이 접근 방식을 가이드할 때

예시:
```markdown
## 코드 리뷰 프로세스

1. 코드 구조와 조직을 분석하십시오.
2. 잠재적인 버그나 엣지 케이스를 확인하십시오.
3. 가독성과 유지보수성 향상을 위한 개선 사항을 제안하십시오.
4. 프로젝트 컨벤션 준수 여부를 확인하십시오.
```

**중간 자유도** (의사코드 또는 매개변수가 있는 스크립트):

다음의 경우에 사용하십시오:
* 선호되는 패턴이 존재할 때
* 약간의 변형이 허용될 때
* 설정(Configuration)이 동작에 영향을 줄 때

예시:
````markdown
## 보고서 생성

이 템플릿을 사용하고 필요에 따라 커스터마이징하십시오:

```python
def generate_report(data, format="markdown", include_charts=True):
    # 데이터 처리
    # 지정된 형식으로 출력 생성
    # 선택적으로 시각화 포함
```
````

**낮은 자유도** (구체적인 스크립트, 매개변수가 거의 없거나 없음):

다음의 경우에 사용하십시오:
* 작업이 취약하고 오류가 발생하기 쉬울 때
* 일관성이 매우 중요할 때
* 특정 순서를 반드시 따라야 할 때

예시:
````markdown
## 데이터베이스 마이그레이션

반드시 이 스크립트를 실행하십시오:

```bash
python scripts/migrate.py --verify --backup
```

명령어를 수정하거나 추가 플래그를 더하지 마십시오.
````

**비유:** Claude를 길을 탐험하는 로봇이라고 생각해보십시오:

* **양쪽이 절벽인 좁은 다리**: 앞으로 나아갈 수 있는 안전한 길은 오직 하나뿐입니다. 구체적인 가드레일과 정확한 지침을 제공하십시오 (낮은 자유도). 예: 정확한 순서대로 실행되어야 하는 데이터베이스 마이그레이션.
* **위험 요소가 없는 넓은 벌판**: 많은 길이 성공으로 이어집니다. 일반적인 방향을 제시하고 Claude가 최선의 경로를 찾도록 신뢰하십시오 (높은 자유도). 예: 컨텍스트에 따라 최선의 접근 방식이 결정되는 코드 리뷰.

### 사용할 모든 모델에서 테스트하십시오

기술은 모델에 추가되는 기능과 같으므로, 그 효과는 기반이 되는 모델에 따라 달라집니다. 여러분이 사용하려는 모든 모델에서 기술을 테스트하십시오.

**모델별 테스트 고려 사항**:

* **Claude Haiku** (빠름, 경제적): 기술이 충분한 가이드를 제공하는가?
* **Claude Sonnet** (균형 잡힘): 기술이 명확하고 효율적인가?
* **Claude Opus** (강력한 추론): 기술이 과도한 설명을 하지는 않는가?

Opus에서 완벽하게 작동하는 것이 Haiku에서는 더 자세한 설명이 필요할 수도 있습니다. 여러 모델에서 기술을 사용할 계획이라면, 모든 모델에서 잘 작동하는 지침을 작성하는 것을 목표로 하십시오.

## 기술 구조

> [!NOTE]
> **YAML 프론트매터**: `SKILL.md` 프론트매터에는 두 개의 필드가 필수입니다:
>
> * `name` - 기술의 사람이 읽을 수 있는 이름 (최대 64자)
> * `description` - 기술이 무엇을 하는지, 언제 사용하는지에 대한 한 줄 설명 (최대 1024자)
>
> 완전한 기술 구조 세부 사항은 [Skills overview](https://platform.claude.com/docs/en/agents-and-tools/agent-skills/overview#skill-structure)를 참조하십시오.

### 명명 규칙

기술을 더 쉽게 참조하고 논의할 수 있도록 일관된 명명 패턴을 사용하십시오. 기술이 제공하는 활동이나 기능을 명확하게 설명할 수 있도록 **동명사 형태** (동사 + -ing)를 기술 이름에 사용하는 것을 권장합니다.

**좋은 명명 예시 (동명사 형태)**:

* "Processing PDFs" (PDF 처리하기)
* "Analyzing spreadsheets" (스프레드시트 분석하기)
* "Managing databases" (데이터베이스 관리하기)
* "Testing code" (코드 테스트하기)
* "Writing documentation" (문서 작성하기)

**허용 가능한 대안**:

* 명사구: "PDF Processing", "Spreadsheet Analysis"
* 동작 지향: "Process PDFs", "Analyze Spreadsheets"

**피해야 할 것**:

* 모호한 이름: "Helper", "Utils", "Tools"
* 너무 일반적인 이름: "Documents", "Data", "Files"
* 기술 컬렉션 내에서 일관되지 않은 패턴

일관된 명명은 다음을 용이하게 합니다:

* 문서나 대화에서 기술을 참조하기 쉬워짐
* 한눈에 기술이 무엇을 하는지 이해하기 쉬워짐
* 여러 기술을 정리하고 검색하기 쉬워짐
* 전문적이고 응집력 있는 기술 라이브러리 유지

### 효과적인 설명(Description) 작성하기

`description` 필드는 기술을 발견할 수 있게 해주며, 기술이 무엇을 하는지와 언제 사용하는지를 모두 포함해야 합니다.

> [!WARNING]
> **항상 3인칭으로 작성하십시오**. 설명은 시스템 프롬프트에 주입되므로, 일관되지 않은 관점은 검색 문제를 일으킬 수 있습니다.
>
> * **좋음:** "Excel 파일을 처리하고 보고서를 생성함"
> * **피함:** "귀하의 Excel 파일 처리를 도와드릴 수 있습니다"
> * **피함:** "이 기술을 사용하여 Excel 파일을 처리할 수 있습니다"

**구체적이어야 하며 핵심 용어를 포함하십시오**. 기술이 하는 일과 언제 사용해야 하는지에 대한 특정 트리거/컨텐츠를 모두 포함하십시오.

각 기술은 정확히 하나의 설명 필드를 갖습니다. 설명은 기술 선택에 있어 결정적입니다. Claude는 수백 개의 사용 가능한 기술 중에서 적절한 기술을 선택하기 위해 이를 사용합니다. 설명은 Claude가 이 기술을 언제 선택해야 하는지 알 수 있을 만큼 충분한 세부 정보를 제공해야 하며, `SKILL.md`의 나머지 부분은 상세 구현 내용을 제공합니다.

효과적인 예시:

**PDF 처리 기술:**
```yaml
description: PDF 파일에서 텍스트와 표를 추출하고, 양식을 채우고, 문서를 병합함. PDF 파일로 작업하거나 사용자가 PDF, 양식 또는 문서 추출을 언급할 때 사용함.
```

**Excel 분석 기술:**
```yaml
description: Excel 스프레드시트를 분석하고, 피벗 테이블을 만들고, 차트를 생성함. Excel 파일, 스프레드시트, 표 형식 데이터 또는 .xlsx 파일을 분석할 때 사용함.
```

**Git 커밋 헬퍼 기술:**
```yaml
description: git diff를 분석하여 설명적인 커밋 메시지를 생성함. 사용자가 커밋 메시지 작성을 위해 도움을 요청하거나 스테이징된 변경 사항을 검토할 때 사용함.
```

다음과 같이 모호한 설명은 피하십시오:
```yaml
description: 문서 작성을 도와줌
description: 데이터를 처리함
description: 파일로 작업을 수행함
```

### 점진적 노출(Progressive Disclosure) 패턴

`SKILL.md`는 온보딩 가이드의 목차처럼, 필요에 따라 Claude를 상세 자료로 안내하는 개요 역할을 합니다. 점진적 노출이 작동하는 방식에 대한 설명은 개요의 [How Skills work](https://platform.claude.com/docs/en/agents-and-tools/agent-skills/overview#how-skills-work)를 참조하십시오.

**실질적 가이드:**

* 최적의 성능을 위해 `SKILL.md` 본문은 500줄 이하로 유지하십시오.
* 이 제한에 도달하면 콘텐츠를 별도의 파일로 분리하십시오.
* 지침, 코드 및 리소스를 효과적으로 정리하려면 아래의 패턴을 사용하십시오.

#### 시나리오: 단순함에서 복잡함으로

기본적인 기술은 메타데이터와 지침을 포함하는 `SKILL.md` 파일 하나로 시작합니다.

기술이 성장함에 따라, Claude가 필요할 때만 로드할 수 있도록 추가 콘텐츠를 묶을 수 있습니다.

완성된 기술 디렉토리 구조는 다음과 같을 수 있습니다:

```
pdf/
├── SKILL.md              # 주요 지침 (트리거될 때 로드됨)
├── FORMS.md              # 양식 채우기 가이드 (필요할 때 로드됨)
├── reference.md          # API 참조 (필요할 때 로드됨)
├── examples.md           # 사용 예시 (필요할 때 로드됨)
└── scripts/
    ├── analyze_form.py   # 유틸리티 스크립트 (실행됨, 로드되지 않음)
    ├── fill_form.py      # 양식 채우기 스크립트
    └── validate.py       # 검증 스크립트
```

#### 패턴 1: 참조를 포함한 상위 수준 가이드

````markdown
---
name: PDF Processing
description: PDF 파일에서 텍스트와 표를 추출하고, 양식을 채우고, 문서를 병합함. PDF 파일로 작업하거나 사용자가 PDF, 양식 또는 문서 추출을 언급할 때 사용함.
---

# PDF 처리 (PDF Processing)

## 빠른 시작

pdfplumber로 텍스트를 추출하십시오:
```python
import pdfplumber
with pdfplumber.open("file.pdf") as pdf:
    text = pdf.pages[0].extract_text()
```

## 고급 기능

**양식 채우기**: 전체 가이드는 [FORMS.md](FORMS.md)를 참조하십시오.
**API 참조**: 모든 메서드는 [REFERENCE.md](REFERENCE.md)를 참조하십시오.
**예시**: 일반적인 패턴은 [EXAMPLES.md](EXAMPLES.md)를 참조하십시오.
````

Claude는 필요할 때만 `FORMS.md`, `REFERENCE.md` 또는 `EXAMPLES.md`를 로드합니다.

#### 패턴 2: 도메인별 구성

여러 도메인이 포함된 기술의 경우, 관련 없는 컨텍스트가 로드되는 것을 피하기 위해 도메인별로 콘텐츠를 구성하십시오. 사용자가 매출 지표에 대해 물어볼 때, Claude는 재무나 마케팅 데이터가 아닌 매출 관련 스키마만 읽으면 됩니다. 이는 토큰 사용량을 낮게 유지하고 컨텍스트를 집중시킵니다.

```
bigquery-skill/
├── SKILL.md (개요 및 탐색)
└── reference/
    ├── finance.md (매출, 청구 지표)
    ├── sales.md (기회, 파이프라인)
    ├── product.md (API 사용량, 기능)
    └── marketing.md (캠페인, 어트리뷰션)
```

#### 패턴 3: 조건부 세부 정보

기본 콘텐츠를 보여주고 고급 콘텐츠로 링크를 연결하십시오:

```markdown
# DOCX 처리

## 문서 생성

새 문서에는 docx-js를 사용하십시오. [DOCX-JS.md](DOCX-JS.md)를 참조하십시오.

## 문서 편집

단순한 편집의 경우 XML을 직접 수정하십시오.
```

### 깊게 중첩된 참조 피하기

참조된 파일에서 또 다른 파일이 참조될 때, Claude는 파일을 부분적으로만 읽을 수 있습니다. 중첩된 참조가 있을 경우 Claude는 전체 파일을 읽는 대신 `head -100`과 같은 명령을 사용하여 컨텐츠를 미리 볼 수 있으며, 이로 인해 정보가 불완전해질 수 있습니다.

**참조는 SKILL.md에서 한 단계 깊이까지만 유지하십시오**. Claude가 필요할 때 완벽한 파일을 읽을 수 있도록 모든 참조 파일은 `SKILL.md`에서 직접 링크되어야 합니다.

**나쁜 예: 너무 깊음**:
```markdown
# SKILL.md
[advanced.md](advanced.md) 참조...

# advanced.md
[details.md](details.md) 참조...

# details.md
여기에 실제 정보가 있음...
```

**좋은 예: 한 단계 깊이**:
```markdown
# SKILL.md

**기본 사용법**: [SKILL.md에 있는 지침]
**고급 기능**: [advanced.md](advanced.md) 참조
**API 참조**: [reference.md](reference.md) 참조
**예시**: [examples.md](examples.md) 참조
```

### 긴 참조 파일은 목차로 구성하십시오

100줄이 넘는 참조 파일의 경우 상단에 목차를 포함하십시오. 이를 통해 Claude는 부분 읽기로 미리 볼 때도 사용 가능한 정보의 전체 범위를 파악할 수 있습니다.

Claude는 필요에 따라 전체 파일을 읽거나 특정 섹션으로 이동할 수 있습니다.

이러한 파일 시스템 기반 아키텍처가 어떻게 점진적 노출을 가능하게 하는지에 대한 자세한 내용은 아래 고급 섹션의 [런타임 환경](#런타임-환경)을 참조하십시오.

## 워크플로우 및 피드백 루프

### 복잡한 작업에는 워크플로우를 사용하십시오

복잡한 작업을 명확하고 순차적인 단계로 나누십시오. 특히 복잡한 워크플로우의 경우, Claude가 응답에 복사하여 진행 상황에 따라 체크할 수 있는 체크리스트를 제공하십시오.

**예시 1: 연구 요약 워크플로우** (코드가 없는 기술):
````markdown
## 연구 요약 워크플로우

이 체크리스트를 복사하여 진행 상황을 추적하십시오:

```
연구 진행 상황:
- [ ] 1단계: 모든 소스 문서 읽기
- [ ] 2단계: 핵심 주제 식별
- [ ] 3단계: 주장 교차 검증
- [ ] 4단계: 구조화된 요약 생성
- [ ] 5단계: 인용 확인
```

**1단계: 모든 소스 문서 읽기**
`sources/` 디렉토리의 각 문서를 검토하십시오. 주요 논거와 뒷받침하는 증거를 기록하십시오.

... (중략) ...
````

**예시 2: PDF 양식 채우기 워크플로우** (코드가 있는 기술):
````markdown
## PDF 양식 채우기 워크플로우

이 체크리스트를 복사하고 완료한 항목을 체크하십시오:

```
작업 진행 상황:
- [ ] 1단계: 양식 분석 (analyze_form.py 실행)
- [ ] 2단계: 필드 매핑 생성 (fields.json 편집)
- [ ] 3단계: 매핑 검증 (validate_fields.py 실행)
- [ ] 4단계: 양식 채우기 (fill_form.py 실행)
- [ ] 5단계: 출력 결과 확인 (verify_output.py 실행)
```
... (중략) ...
````

명확한 단계는 Claude가 중요한 검증 단계를 건너뛰는 것을 방지합니다. 체크리스트는 Claude와 여러분 모두가 다단계 워크플로우의 진행 상황을 추적하도록 도와줍니다.

### 피드백 루프 구현

**일반적인 패턴**: 검증기(validator) 실행 → 오류 수정 → 반복

이 패턴은 출력 품질을 크게 향상시킵니다.

**예시 1: 스타일 가이드 준수** (코드가 없는 기술):
```markdown
## 컨텐츠 리뷰 프로세스

1. STYLE_GUIDE.md의 가이드라인에 따라 컨텐츠 초안을 작성하십시오.
2. 체크리스트를 대조하여 리뷰하십시오:
   - 용어의 일관성 확인
   - 예시가 표준 형식을 따르는지 확인
   - 모든 필수 섹션이 포함되어 있는지 확인
3. 문제가 발견된 경우:
   - 특정 섹션 참조와 함께 각 문제를 기록하십시오.
   - 컨텐츠를 수정하십시오.
   - 체크리스트를 다시 리뷰하십시오.
4. 모든 요구 사항이 충족되었을 때만 진행하십시오.
5. 문서를 확정하고 저장하십시오.
```

**예시 2: 문서 편집 프로세스** (코드가 있는 기술):
```markdown
## 문서 편집 프로세스

1. `word/document.xml`을 수정하십시오.
2. **즉시 검증**: `python ooxml/scripts/validate.py unpacked_dir/`를 실행하십시오.
3. 검증이 실패한 경우:
   - 오류 메시지를 자세히 검토하십시오.
   - XML의 문제를 수정하십시오.
   - 검증을 다시 실행하십시오.
4. **검증이 통과된 경우에만 진행하십시오**.
5. 리빌드: `python ooxml/scripts/pack.py unpacked_dir/ output.docx`
6. 결과 문서를 테스트하십시오.
```

검증 루프는 오류를 초기에 잡아냅니다.

## 평가 및 반복

### 평가를 먼저 구축하십시오

**방대한 문서를 작성하기 전에 먼저 평가(evaluation)를 만드십시오.** 이를 통해 여러분의 기술이 상상 속의 문제가 아니라 실제 문제를 해결하도록 보장할 수 있습니다.

**평가 주도 개발(Evaluation-driven development):**
1. **공백 식별**: 기술 없이 Claude를 대표적인 태스크에 실행해 보십시오. 구체적인 실패 사례나 누락된 컨텍스트를 기록하십시오.
2. **평가 생성**: 이러한 공백을 테스트하는 세 가지 시나리오를 만드십시오.
3. **기준선(Baseline) 설정**: 기술 없이 Claude의 성능을 측정하십시오.
4. **최소한의 지침 작성**: 공백을 해결하고 평가를 통과하기에 충분한 내용만 작성하십시오.
5. **반복**: 평가를 실행고 기준선과 비교하며 개선하십시오.

이러한 접근 방식은 실현되지 않을 수도 있는 요구 사항을 예상하는 대신 실제 문제를 해결하도록 도와줍니다.

### Claude와 함께 기술을 반복적으로 개발하십시오

가장 효과적인 기술 개발 프로세스는 Claude 자신을 참여시키는 것입니다. 한 Claude 인스턴스("Claude A")와 협력하여 다른 인스턴스("Claude B")가 사용할 기술을 만드십시오. Claude A는 지침을 설계하고 다듬는 것을 돕고, Claude B는 실제 태스크에서 이를 테스트합니다. 이는 Claude 모델이 효과적인 에이전트 지침을 작성하는 방법과 에이전트에게 어떤 정보가 필요한지를 모두 잘 이해하고 있기 때문에 가능합니다.

**새로운 기술 생성:**
1. **기술 없이 태스크 완료**: 일반적인 프롬프팅으로 Claude A와 함께 문제를 해결하십시오. 작업하면서 자연스럽게 컨텍스트를 제공하고, 선호도를 설명하며, 절차적 지식을 공유하게 될 것입니다. 반복적으로 제공하는 정보가 무엇인지 주목하십시오.
2. **재사용 가능한 패턴 식별**: 태스크 완료 후, 유사한 미래 태스크에 유용할 정보가 무엇인지 식별하십시오.
3. **Claude A에게 기술 생성을 요청하십시오**: "우리가 방금 사용한 이 BigQuery 분석 패턴을 캡처하는 기술(Skill)을 만들어줘. 테이블 스키마, 명명 규칙, 그리고 테스트 계정을 필터링하는 규칙을 포함해줘."
4. **간결함 검토**: Claude A가 불필요한 설명을 추가하지 않았는지 확인하십시오. "승률(win rate)이 무엇인지에 대한 설명은 빼줘 - Claude는 이미 알고 있어."라고 요청하십시오.
5. **정보 아키텍처 개선**: Claude A에게 컨텐츠를 더 효과적으로 정리해달라고 요청하십시오. 예를 들어 "테이블 스키마는 별도의 참조 파일에 있도록 정리해줘. 나중에 테이블을 더 추가할 수도 있으니까."라고 할 수 있습니다.
6. **유사 태스크에서 테스트**: 관련 유즈케이스에 대해 Claude B(기술이 로드된 새로운 인스턴스)를 사용하여 기술을 테스트하십시오. Claude B가 올바른 정보를 찾는지, 규칙을 정확히 적용하는지, 작업을 성공적으로 수행하는지 관찰하십시오.
7. **관찰에 기반한 반복**: Claude B가 어려움을 겪거나 무언가를 놓치면 구체적인 내용과 함께 Claude A에게 돌아가십시오. "Claude가 이 기술을 사용할 때 Q4 날짜 필터링을 잊었어. 날짜 필터링 패턴에 대한 섹션을 추가해야 할까?"

**기존 기술 반복 개선:**
동일한 계층적 패턴이 기술 개선 시에도 이어집니다. 여러분은 다음을 오갑니다:
* **Claude A와 작업** (기술 개선을 돕는 전문가)
* **Claude B와 테스트** (실제 작업을 수행하기 위해 기술을 사용하는 에이전트)
* **Claude B의 동작 관찰** 및 Claude A에게 인사이트 전달

이러한 observe-refine-test 사이클을 계속하십시오. 각 반복은 가정이 아닌 실제 에이전트의 동작을 기반으로 기술을 향상시킵니다.

### Claude가 기술을 어떻게 탐색하는지 관찰하십시오

기술을 반복 개선할 때, Claude가 실제로 어떻게 기술을 사용하는지 주의를 기울이십시오:
* **예상치 못한 탐색 경로**: Claude가 여러분이 예상하지 못한 순서로 파일을 읽습니까? 이는 여러분의 구조가 생각만큼 직관적이지 않음을 의미할 수 있습니다.
* **연결 누락**: Claude가 중요한 파일에 대한 참조를 따르지 못합니까? 링크를 더 명시적이거나 눈에 띄게 할 필요가 있습니다.
* **특정 섹션에 대한 과도한 의존**: Claude가 반복적으로 동일한 파일을 읽는다면, 해당 컨텐츠를 메인 `SKILL.md`에 넣는 것을 고려하십시오.
* **무시된 컨텐츠**: Claude가 번들로 제공된 파일에 한 번도 접근하지 않는다면, 불필요하거나 메인 지침에서 제대로 안내되지 않은 것일 수 있습니다.

## 피해야 할 안티 패턴

### Windows 스타일 경로 피하기
파일 경로에는 윈도우에서도 항상 슬래시('/')를 사용하십시오. 유닉스 스타일 경로는 모든 플랫폼에서 작동하지만, 윈도우 스타일 경로는 유닉스 시스템에서 오류를 일으킵니다.

### 너무 많은 옵션 제공 피하기
필요하지 않다면 여러 접근 방식을 제시하지 마십시오. 기본값을 제공하되 비상구(escape hatch)만 열어두십시오.
````markdown
**나쁜 예: 너무 많은 선택지**: "pypdf를 쓰거나, pdfplumber를 쓰거나, PyMuPDF를 쓰거나..."
**좋은 예: 기본값 제공**: "텍스트 추출에는 pdfplumber를 사용하십시오. OCR이 필요한 스캔된 PDF의 경우 pytesseract를 사용하십시오."
````

## 고급: 실행 가능한 코드가 포함된 기술

### 해결하십시오, 떠넘기지 마십시오
기술을 위한 스크립트를 작성할 때, 에러 처리를 Claude가 해결하도록 떠넘기지 말고 직접 처리하십시오.
구성 매개변수(Ousterhout's law에 따른 "voodoo constants" 방지)도 정당화하고 문서화해야 합니다. 여러분이 적절한 값을 모른다면 Claude가 어떻게 결정하겠습니까?

### 유틸리티 스크립트 제공
Claude가 직접 스크립트를 작성할 수 있더라도, 미리 만들어진 스크립트는 다음과 같은 장점이 있습니다:
* 생성된 코드보다 안정적임
* 토큰 절약 (컨텍스트에 코드를 포함할 필요 없음)
* 시간 절약 (코드 생성 시간이 필요 없음)
* 사용 간 일관성 보장

스크립트를 실행해야 하는지, 아니면 참조용으로 읽어야 하는지 지침에서 명확히 하십시오.

### 시각적 분석 활용
입력을 이미지로 렌더링할 수 있는 경우 Claude가 이를 분석하게 하십시오. Claude의 비전 능력은 레이아웃과 구조를 이해하는 데 도움이 됩니다.

### 검증 가능한 중간 출력 생성
Claude가 복잡한 태스크를 수행할 때 실수를 할 수 있습니다. "계획-검증-실행" 패턴은 Claude가 먼저 구조화된 형식으로 계획을 세우게 하고, 실행 전에 스크립트로 그 계획을 검증함으로써 오류를 조기에 잡아냅니다 (분석 → **계획 파일 생성** → **계획 검증** → 실행 → 확인).

### 런타임 환경
기술은 파일 시스템 접근 권한, bash 명령어, 코드 실행 능력을 갖춘 코드 실행 환경에서 실행됩니다.

**여러분의 작성에 미치는 영향:**

* **파일 경로가 중요합니다**: Claude는 기술 디렉토리를 파일 시스템처럼 탐색합니다.
* **파일 이름을 설명적으로 지으십시오**: `doc2.md`가 아니라 `form_validation_rules.md`와 같이 내용을 나타내는 이름을 사용하십시오.
* **발견을 위해 구성하십시오**: 도메인이나 기능별로 디렉토리를 구조화하십시오.
* **포괄적인 리소스를 묶으십시오**: 완벽한 API 문서, 방대한 예시, 대규모 데이터셋을 포함할 수 있습니다. 접근되기 전까지는 컨텍스트 토큰을 소모하지 않습니다.
* **결정론적 작업에는 스크립트를 선호하십시오**: Claude에게 검증 코드를 생성하라고 하는 대신 `validate_form.py`를 작성하십시오.
* **실행 의도를 명확히 하십시오**: "필드 추출을 위해 `analyze_form.py`를 실행"(실행) vs "`analyze_form.py`에서 추출 알고리즘 참조"(참조용 읽기).

### MCP 도구 참조
기술이 MCP(Model Context Protocol) 도구를 사용하는 경우, 항상 정규화된 도구 이름(`ServerName:tool_name`)을 사용하여 "도구를 찾을 수 없음" 오류를 피하십시오. 서버 접두사가 없으면 Claude는 특히 여러 MCP 서버가 사용 가능할 때 도구를 찾지 못할 수 있습니다.

### 도구가 설치되어 있다고 가정하지 마십시오
패키지가 사용 가능하다고 가정하지 마십시오. `SKILL.md`에 필수 패키지를 명시하고 [code execution tool documentation](https://platform.claude.com/docs/en/docs/agents-and-tools/tool-use/code-execution-tool)에서 확인하십시오.
