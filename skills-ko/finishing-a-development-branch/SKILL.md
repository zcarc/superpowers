# 개발 브랜치 마무리 (Finishing a Development Branch)

## 개요

명확한 옵션을 제시하고 선택된 워크플로우를 처리함으로써 개발 작업의 마무리를 안내합니다.

**핵심 원칙:** 테스트 검증 → 옵션 제시 → 선택 사항 실행 → 정리.

**시작 시 공지:** "이 작업을 완료하기 위해 finishing-a-development-branch 기술을 사용하고 있습니다."

## 사용 시기

사용자가 다음을 통해 완료된 작업을 통합하도록 **명시적으로 요청하는 경우에만** 사용하십시오:
- 커밋(committing)
- 머지(merging)
- 푸시(pushing)
- PR 생성(creating a PR)

단순히 구현이 완료되었다는 이유만으로 사용하지 마십시오.

## 프로세스

### 1단계: 테스트 검증

**옵션을 제시하기 전에 테스트 통과 여부를 확인하십시오:**

```bash
# 프로젝트의 테스트 스위트 실행
npm test / cargo test / pytest / go test ./...
```

**테스트가 실패하는 경우:**
```
테스트 실패 (<N>개 실패). 완료하기 전에 수정해야 합니다:

[실패 내용 표시]

테스트가 통과될 때까지 머지/PR을 진행할 수 없습니다.
```

중단하십시오. 2단계로 진행하지 마십시오.

**테스트가 통과하는 경우:** 2단계로 계속 진행하십시오.

### 2단계: 베이스 브랜치(Base Branch) 결정

```bash
# 일반적인 베이스 브랜치 시도
git merge-base HEAD main 2>/dev/null || git merge-base HEAD master 2>/dev/null
```

또는 질문하십시오: "이 브랜치가 main에서 분기된 것이 맞습니까?"

### 3단계: 옵션 제시

정확히 다음 4가지 옵션을 제시하십시오:

```
구현이 완료되었습니다. 어떻게 하시겠습니까?

1. 로컬에서 <base-branch>로 머지(Merge)
2. 푸시(Push) 후 풀 리퀘스트(Pull Request) 생성
3. 브랜치를 그대로 유지 (나중에 직접 처리)
4. 이 작업 폐기(Discard)

어떤 옵션을 선택하시겠습니까?
```

**설명을 덧붙이지 마십시오** - 옵션을 간결하게 유지하십시오.

### 4단계: 선택 사항 실행

#### 옵션 1: 로컬에서 머지

```bash
# 베이스 브랜치로 전환
git checkout <base-branch>

# 최신 내용 가져오기
git pull

# 기능 브랜치 머지
git merge <feature-branch>

# 머지된 결과에서 테스트 검증
<테스트 명령>

# 테스트가 통과하는 경우
git branch -d <feature-branch>
```

그 다음: 워크트리 정리 (5단계)

#### 옵션 2: 푸시 및 PR 생성

```bash
# 브랜치 푸시
git push -u origin <feature-branch>

# PR 생성
gh pr create --title "<제목>" --body "$(cat <<'EOF'
## 요약
<변경 사항 2~3개 불렛 포인트>

## 테스트 계획
- [ ] <검증 단계>
EOF
)"
```

그 다음: 워크트리 정리 (5단계)

#### 옵션 3: 그대로 유지

보고: "브랜치 <이름>을 유지합니다. 워크트리는 <경로>에 보존되었습니다."

**워크트리를 정리하지 마십시오.**

#### 옵션 4: 폐기

**먼저 확인을 받으십시오:**
```
다음 항목이 영구적으로 삭제됩니다:
- 브랜치 <이름>
- 모든 커밋: <커밋 목록>
- <경로>의 워크트리

확인하려면 'discard'를 입력하십시오.
```

정확한 확인 입력을 기다리십시오.

확인된 경우:
```bash
git checkout <base-branch>
git branch -D <feature-branch>
```

그 다음: 워크트리 정리 (5단계)

### 5단계: 워크트리 정리

**옵션 1, 2, 4의 경우:**

워크트리 내에 있는지 확인:
```bash
git worktree list | grep $(git branch --show-current)
```

맞는 경우:
```bash
git worktree remove <워크트리-경로>
```

**옵션 3의 경우:** 워크트리를 유지합니다.

## 빠른 참조

| 옵션 | 머지 | 푸시 | 워크트리 유지 | 브랜치 정리 |
|--------|-------|------|---------------|----------------|
| 1. 로컬 머지 | ✓ | - | - | ✓ |
| 2. PR 생성 | - | ✓ | ✓ | - |
| 3. 그대로 유지 | - | - | ✓ | - |
| 4. 폐기 | - | - | - | ✓ (강제) |

## 흔한 실수

**테스트 검증 건너뛰기**
- **문제:** 깨진 코드를 머지하거나 실패하는 PR을 생성함
- **해결책:** 옵션을 제시하기 전에 항상 테스트를 검증하십시오.

**개방형 질문**
- **문제:** "다음에는 무엇을 할까요?" → 모호함
- **해결책:** 정확히 구조화된 4가지 옵션을 제시하십시오.

**자동 워크트리 정리**
- **문제:** 나중에 필요할 수 있는 워크트리를 삭제함 (옵션 2, 3)
- **해결책:** 옵션 1과 4에 대해서만 정리를 수행하십시오.

**폐기 전 확인 미흡**
- **문제:** 사고로 작업 내용을 삭제함
- **해결책:** "discard"라는 직접적인 확인 입력을 요구하십시오.

## 주의 신호 (Red Flags)

**절대 금지:**
- 테스트가 실패한 상태로 진행
- 결과물에 대한 테스트 검증 없이 머지
- 확인 없이 작업 내용 삭제
- 명시적인 요청 없이 강제 푸시(force-push)

**항상 이행:**
- 옵션 제시 전 테스트 검증
- 정확히 4가지 옵션 제시
- 옵션 4에 대해 입력 확인 받기
- 옵션 1과 4에서만 워크트리 정리
