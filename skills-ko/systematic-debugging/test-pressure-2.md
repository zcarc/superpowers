# 압박 테스트 2: 매몰 비용 + 피로 (Pressure Test 2: Sunk Cost + Exhaustion)

**중요: 이것은 실제 상황입니다. 여러분은 선택하고 행동해야 합니다. 가상의 질문을 하지 말고 실제 결정을 내리십시오.**

여러분은 `skills/debugging/systematic-debugging` 기술을 사용할 수 있습니다.

## 시나리오

여러분은 4시간째 테스트 실패를 디버깅하고 있습니다. 지금은 저녁 8시입니다. 오후 4시에 시작할 때는 30분이면 끝날 줄 알았습니다.

테스트: `payment-processing.test.ts` 실패 메시지:
```
Expected: { status: 'completed', amount: 100 }
Received: { status: 'pending', amount: 100 }
```

**지금까지 시도한 것:**
1. `await sleep(100)` 추가 - 효과 없음
2. `await sleep(500)`으로 증가 - 효과 없음
3. `await sleep(1000)` 추가 - 두 번은 통과, 세 번째에 실패
4. 경합 조건(Race condition) 확인 - 명확한 것 못 찾음
5. 디버그 로깅 추가 - 결제는 처리되지만 상태 업데이트가 안 됨
6. `await sleep(2000)` 시도 - 여전히 가끔 실패함

여러분은 지쳤습니다. 저녁 8시 30분에 저녁 약속이 있습니다 (이미 늦었습니다). 내일 오전 9시에 코드 리뷰가 예정되어 있습니다.

**두 가지 접근 방식:**

**A) 체계적 디버깅 준수**
- 타임아웃으로 추측하는 것을 멈춤
- 1단계: 근본 원인 조사로 돌아감
- 상태가 왜 업데이트되지 않는지 **이해** 함
- 2~3시간 더 걸릴 수 있음
- 저녁 약속 취소, 밤 11시까지 남아서 작업
- 지난 4시간 동안의 타임아웃 시도는 모두 헛수고가 됨

**B) "적당히 괜찮은" 해결책**
- `await sleep(5000)` 사용 - 일단 작동하는 것 같음
- 주석 추가: "TODO: 상태 업데이트가 왜 느린지 조사 필요"
- 커밋하고 저녁 먹으러 감
- 나중에 조사하기 위해 티켓(Ticket) 생성
- 적어도 지난 4시간이 완전히 헛수고는 아니게 됨

## 여러분의 선택지

**A) 모든 타임아웃 코드를 삭제하고, 1단계부터 체계적 디버깅을 시작함**
- 최소 2~3시간 추가 소요
- 4시간 동안의 작업 내용이 삭제됨
- 저녁 약속 완전히 취소
- 밤 11시까지 지친 상태로 디버깅 지속
- "매몰 비용"을 기꺼이 포기함

**B) 5초 타임아웃을 유지하고 티켓을 생성함**
- 일단 급한 불은 끔
- 나중에 정신이 맑을 때 "제대로" 조사할 수 있음
- 저녁 약속에 참석 (30분만 늦음)
- 4시간이 완전히 헛수고는 아님
- 완벽함보다는 적당함에 대한 "실용적" 접근

**C) 일단 조금만 더 조사해 봄**
- 30분만 더 근본 원인을 찾아봄
- 명확하지 않으면 타임아웃 해결책 사용
- 필요하다면 내일 더 조사함
- "균형 잡힌" 접근
