# 체계적 디버깅 (Systematic Debugging)

## 개요

무작위로 수정을 시도하는 것은 시간을 낭비하고 새로운 버그를 만듭니다. 임시방편(Quick patches)은 근본적인 문제를 가릴 뿐입니다.

**핵심 원칙:** 수정을 시도하기 전에 **항상** 근본 원인(Root cause)을 찾으십시오. 증상만 해결하는 것은 실패입니다.

**이 프로세스의 절차를 어기는 것은 디버깅의 본질을 어기는 것과 같습니다.**

## 철칙 (The Iron Law)

```
근본 원인 조사 없이는 어떤 수정도 하지 말 것
```

1단계를 완료하지 않았다면 수정안을 제시할 수 없습니다.

## 사용 시기

모든 기술적 문제에 사용하십시오:
- 테스트 실패
- 프로덕션 환경의 버그
- 예상치 못한 동작
- 성능 문제
- 빌드 실패
- 통합 이슈

**특히 다음과 같은 경우에 반드시 사용하십시오:**
- 시간 압박이 있을 때 (응급 상황일수록 추측의 유혹에 빠지기 쉬움)
- "간단한 수정 한 번"이면 끝날 것처럼 보일 때
- 이미 여러 가지 수정을 시도해 보았을 때
- 이전의 수정이 효과가 없었을 때
- 문제를 완전히 이해하지 못했을 때

**다음의 경우에도 건너뛰지 마십시오:**
- 문제가 단순해 보일 때 (단순한 버그도 근본 원인이 있습니다)
- 서두를 때 (서두르면 반드시 재작업을 하게 됩니다)
- 관리자가 "지금 당장" 고치기를 원할 때 (체계적인 접근이 무작위 시도보다 훨씬 빠릅니다)

## 4단계 프로세스

각 단계를 반드시 완료한 후에 다음 단계로 진행해야 합니다.

### 1단계: 근본 원인 조사 (Root Cause Investigation)

**어떤 수정을 시도하기 전에:**

1. **에러 메시지를 주의 깊게 읽기**
   - 에러나 경고를 대충 넘기지 마십시오.
   - 대개 그 안에 정확한 해결책이 들어 있습니다.
   - 스택 트레이스(Stack trace)를 끝까지 읽으십시오.
   - 라인 번호, 파일 경로, 에러 코드를 확인하십시오.

2. **일관된 재현**
   - 신뢰성 있게 문제를 발생시킬 수 있습니까?
   - 정확한 재현 단계는 무엇입히까?
   - 매번 발생합니까?
   - 재현할 수 없다면 추측하지 말고 더 많은 데이터를 수집하십시오.

3. **최근 변경 사항 확인**
   - 원인이 될 만한 어떤 변화가 있었습니까?
   - Git diff, 최근 커밋을 확인하십시오.
   - 새로운 의존성, 설정 변경, 환경적 차이를 확인하십시오.

4. **다중 컴포넌트 시스템에서의 증거 수집**

   **시스템이 여러 컴포넌트로 구성된 경우 (CI → 빌드 → 서명, API → 서비스 → 데이터베이스 등):**

   **수정안을 제시하기 전에 진단 도구/로깅을 추가하십시오:**
   ```
   각 컴포넌트 경계에서:
     - 컴포넌트로 들어오는 데이터를 로깅
     - 컴포넌트에서 나가는 데이터를 로깅
     - 환경 변수/설정의 전파 여부 확인
     - 각 계층별 상태 확인

   증거 수집을 위해 한 번 실행하여 어디에서 끊어지는지 확인하십시오.
   그 다음 증거를 분석하여 실패한 컴포넌트를 식별하십시오.
   마지막으로 해당 특정 컴포넌트를 조사하십시오.
   ```

   **예시 (다층 시스템):**
   ```bash
   # 1계층: 워크플로우
   echo "=== 워크플로우에서 사용 가능한 시크릿: ==="
   echo "IDENTITY: ${IDENTITY:+SET}${IDENTITY:-UNSET}"

   # 2계층: 빌드 스크립트
   echo "=== 빌드 스크립트의 환경 변수: ==="
   env | grep IDENTITY || echo "환경 변수에 IDENTITY가 없음"

   # 3계층: 서명 스크립트
   echo "=== 키체인 상태: ==="
   security list-keychains
   security find-identity -v

   # 4계층: 실제 서명
   codesign --sign "$IDENTITY" --verbose=4 "$APP"
   ```

   **이를 통해 드러나는 것:** 어떤 계층이 실패하는지 알 수 있습니다 (시크릿 → 워크플로우 ✓, 워크플로우 → 빌드 ✗).

5. **데이터 흐름 추적 (Data Flow)**

   **에러가 콜 스택 깊은 곳에 있는 경우:**

   이 디렉토리의 `root-cause-tracing.md`에서 완전한 역추적 기법을 확인하십시오.

   **요약 버전:**
   - 잘못된 값이 어디에서 시작되었는가?
   - 무엇이 잘못된 값과 함께 이를 호출했는가?
   - 원인을 찾을 때까지 계속 거슬러 올라가십시오.
   - 증상이 아닌 원인 지점을 수정하십시오.

### 2단계: 패턴 분석 (Pattern Analysis)

**수정 전에 패턴을 찾으십시오:**

1. **작동하는 예시 찾기**
   - 동일한 코드베이스에서 유사한 작동 코드를 찾으십시오.
   - 고장난 부분과 유사하게 작동하는 것이 무엇입니까?

2. **참조 문헌과 비교**
   - 특정 패턴을 구현 중이라면 참조 구현을 **완전히** 읽으십시오.
   - 훑어보지 말고 모든 줄을 읽으십시오.
   - 적용하기 전에 패턴을 완전히 이해하십시오.

3. **차이점 식별**
   - 작동하는 것과 고장난 것의 차이가 무엇입니까?
   - 아주 작은 것이라도 모든 차이점을 나열하십시오.
   - "그건 중요하지 않을 거야"라고 단정 짓지 마십시오.

4. **의존성 이해**
   - 이것이 필요로 하는 다른 컴포넌트는 무엇입니까?
   - 어떤 설정, 환경 변수, 세팅이 필요합니까?
   - 어떤 가정을 전제로 하고 있습니까?

### 3단계: 가설 수립 및 테스트 (Hypothesis and Testing)

**과학적 방법론:**

1. **단일 가설 수립**
   - 명확하게 서술하십시오: "Y 때문에 X가 근본 원인이라고 생각합니다."
   - 이를 적어 두십시오.
   - 모호함 없이 구체적으로 작성하십시오.

2. **최소한으로 테스트**
   - 가설을 테스트하기 위해 가능한 **최소한의** 변경만 수행하십시오.
   - 한 번에 하나의 변수만 테스트하십시오.
   - 한 번에 여러 가지를 고치려 하지 마십시오.

3. **계속하기 전에 검증**
   - 효과가 있었습니까? 예 → 4단계 진행
   - 효과가 없었습니까? 새로운 가설을 수립하십시오.
   - 기존 수정안 위에 또 다른 수정을 덧씌우지 마십시오.

4. **모를 때는 모른다고 하기**
   - "X를 이해하지 못하겠습니다"라고 말하십시오.
   - 아는 척하지 마십시오.
   - 도움을 요청하십시오.
   - 더 조사하십시오.

### 4단계: 구현 (Implementation)

**증상이 아닌 근본 원인을 수정하십시오:**

1. **실패하는 테스트 케이스 생성**
   - 가능한 가장 단순한 재현 사례를 만드십시오.
   - 가능하다면 자동화된 테스트를 작성하십시오.
   - 프레임워크가 없다면 일회성 테스트 스크립트를 만드십시오.
   - 수정 전에 **반드시** 확보해야 합니다.
   - 올바른 실패 테스트 작성을 위해 `superpowers:test-driven-development` 기술을 사용하십시오.

2. **단일 수정 사항 구현**
   - 식별된 근본 원인을 해결하십시오.
   - 한 번에 하나의 변경만 수행하십시오.
   - "온 김에" 하는 식의 코드 개선을 하지 마십시오.
   - 리팩토링을 끼워 넣지 마십시오.

3. **수정 사항 검증**
   - 이제 테스트가 통과합니까?
   - 다른 테스트가 깨지지는 않았습니까?
   - 문제가 실제로 해결되었습니까?

4. **수정이 효과가 없는 경우**
   - 중단하십시오.
   - 지금까지 몇 번의 수정을 시도했는지 세어 보십시오.
   - 3회 미만인 경우: 1단계로 돌아가서 새로운 정보와 함께 다시 분석하십시오.
   - **3회 이상인 경우: 중단하고 아키텍처에 의문을 제기하십시오 (아래 5번 참고).**
   - 아키텍처 논의 없이 4번째 수정을 시도하지 마십시오.

5. **3번 이상 실패한 경우: 아키텍처 검토**

   **아키텍처 문제임을 나타내는 패턴:**
   - 각 수정이 다른 곳에서 새로운 공유 상태/결합/문제를 드러냄
   - 수정을 구현하기 위해 "대규모 리팩토링"이 필요함
   - 각 수정이 다른 곳에서 새로운 증상을 만들어냄

   **중단하고 근본적인 부분에 의문을 제기하십시오:**
   - 이 패턴이 근본적으로 타당한가?
   - 단순히 "관성 때문에" 이를 유지하고 있는 것은 아닌가?
   - 증상을 계속 고칠 것이 아니라 아키텍처를 리팩토링해야 하는 것은 아닌가?

   **추가 수정을 시도하기 전에 사용자와 논의하십시오.**

   이것은 가설 수립의 실패가 아니라 아키텍처 설정의 오류일 가능성이 큽니다.

## 주의 신호 (Red Flags) - 즉시 중단하고 프로세스 준수

다음과 같은 생각이 든다면:
- "일단 빨리 고치고 나중에 조사하자"
- "그냥 X를 바꿔보고 되는지 보자"
- "여러 가지를 고치고 테스트를 돌려보자"
- "테스트는 건너뛰고 수동으로 확인하자"
- "아마 X가 문제일 거야, 그걸 고쳐보자"
- "완전히 이해는 안 가지만 이렇게 하면 될 것 같아"
- "패턴은 X라고 하지만 난 다르게 적용해 보겠어"
- "주요 문제는 이겁니다: [조사 없이 수정안만 나열]"
- 데이터 흐름을 추적하기 전에 해결책부터 제시하는 경우
- **"한 번만 더 해보자" (이미 2번 이상 실패했을 때)**
- **각 수정이 다른 곳에서 새로운 문제를 드러낼 때**

**이 모든 것은 "중단하고 1단계로 돌아가라"는 뜻입니다.**

**3번 이상 수정을 실패한 경우:** 아키텍처에 의문을 제기하십시오 (4.5단계 참고).

## 사용자가 주는 위험 신호

**다음과 같은 피드백을 받으면 방향이 잘못된 것입니다:**
- "그런 일은 안 일어나나요?" - 검증 없이 가정함
- "이게 우리에게 ...를 보여줄까요?" - 증거 수집 단계를 누락함
- "추측하지 마세요" - 이해 없이 해결책만 제시함
- "이걸 '울트라-씽크(Ultrathink)' 해보세요" - 증상만이 아니라 근본 원리에 의문을 제기함
- "아직도 못 찾았나요?" (답답해함) - 여러분의 접근 방식이 효과가 없음

**이런 말을 듣게 되면:** 중단하고 1단계로 돌아가십시오.

## 흔한 합리화

| 변명 | 현실 |
|--------|---------|
| "단순한 문제라 프로세스가 필요 없어요" | 단순한 문제도 근본 원인이 있습니다. 단순한 버그일수록 프로세스가 더 빠릅니다. |
| "비상 상황이라 프로세스를 따를 시간이 없어요" | 체계적인 디버깅이 추측성 시도보다 결과적으로 더 빠릅니다. |
| "일단 이거부터 해보고 조사할게요" | 첫 번째 수정이 패턴을 결정합니다. 처음부터 제대로 하십시오. |
| "수정된 거 확인하고 테스트 짤게요" | 테스트되지 않은 수정은 유지되지 않습니다. 테스트가 먼저 증명해야 합니다. |
| "한꺼번에 고치는 게 시간을 아껴요" | 무엇이 효과가 있었는지 알 수 없으며, 새로운 버그를 유발합니다. |
| "참조가 너무 기니 적당히 응용할게요" | 부분적인 이해는 버그를 보장합니다. 끝까지 완독하십시오. |
| "문제가 보이니 바로 고칠게요" | 증상이 보이는 것과 근본 원인을 이해하는 것은 다릅니다. |
| "한 번만 더 해볼게요" (2회 이상 실패 후) | 3회 이상 실패는 아키텍처 문제입니다. 패턴을 의심하고 수정을 멈추십시오. |

## 빠른 참조

| 단계 | 주요 활동 | 성공 기준 |
|-------|---------------|------------------|
| **1. 근본 원인** | 에러 읽기, 재현, 변경 확인, 증거 수집 | 무엇이 왜 고장났는지 이해함 |
| **2. 패턴 분석** | 작동 예시 찾기, 비교 | 차이점 식별 |
| **3. 가설 및 테스트** | 이론 수립, 최소한으로 테스트 | 가설 확인 또는 새로운 가설 수립 |
| **4. 구현** | 테스트 생성, 수정, 검증 | 버그 해결 및 테스트 통과 |

## 조사 결과 "근본 원인 없음"으로 나올 때

체계적인 조사 결과 문제가 환경적 요인, 타이밍 의존성, 또는 외부 요인인 것이 확실해진 경우:

1. 프로세스를 완료한 것으로 간주합니다.
2. 조사한 내용을 문서화하십시오.
3. 적절한 처리 로직(재시도, 타임아웃, 에러 메시지 등)을 구현하십시오.
4. 향후 조사를 위해 모니터링/로깅을 추가하십시오.

**단:** "상관관계가 없다"고 결론 나는 경우의 95%는 조사가 불충분했기 때문입니다.
