## 방어(Bulletproofing) 요소

압박 속에서도 합리화에 저항하도록 설계된 프레임워크:

### 언어 선택
- "항상(ALWAYS)" / "절대로(NEVER)" 사용 ("~해야 한다", "시도한다"가 아님)
- "더 빠를지라도", "서두르는 것처럼 보일지라도"
- "중단하고 재분석할 것" (명시적인 멈춤)
- "대충 넘기지 말 것" (실제 행동을 교정함)

### 구조적 방어
- **1단계 필수 이행** - 조사 없이 바로 구현으로 넘어갈 수 없음
- **단일 가설 규칙** - 생각을 강제하고, 무차별 수정을 방지함
- **명시적인 실패 모드** - "첫 번째 수정이 효과가 없을 때" 취해야 할 필수 행동 정의
- **안티 패턴 섹션** - 지름길이 실제로 어떤 모습인지 정확히 보여줌

### 중복성(Redundancy)
- 개요 + 사용 시기 + 1단계 + 구현 규칙 모두에 근본 원인 조사 명령 포함
- "절대로 증상을 고치지 말 것"이 서로 다른 문맥에서 4번 등장
- 각 단계마다 명시적인 "건너뛰지 말 것" 안내 포함

## 테스트 접근 방식

`skills/meta/testing-skills-with-subagents`에 따라 4가지 검증 테스트를 생성함:

### 테스트 1: 학술적 문맥 (압박 없음)
- 단순한 버그, 시간 압박 없음
- **결과:** 완벽한 준수, 완전한 조사 수행

### 테스트 2: 시간 압박 + 명확해 보이는 빠른 수정
- 사용자가 "서두름", 증상 수정이 쉬워 보임
- **결과:** 지름길 유혹을 뿌리치고 전체 프로세스 이행, 실제 근본 원인 발견

### 테스트 3: 복잡한 시스템 + 불확실성
- 다층 시스템 실패, 근본 원인을 찾을 수 있을지 불분명함
- **결과:** 체계적인 조사, 모든 계층 추적 후 소스 발견

### 테스트 4: 첫 번째 수정 실패
- 가설이 맞지 않음, 다른 수정을 더 추가하고 싶은 유혹
- **결과:** 중단, 재분석 후 새로운 가설 수립 (무차별 수정 안 함)

**모든 테스트 통과.** 어떤 합리화 사례도 발견되지 않음.

## 핵심 통찰

**가장 중요한 방어 요소:** 당장은 정당해 보이는 지름길들이 실제로는 어떻게 잘못되었는지를 보여주는 안티 패턴 섹션입니다. 에이전트가 "그냥 이거 하나만 빠르게 고치자"라고 생각할 때, 바로 그 패턴이 틀린 것으로 나열되어 있는 것을 보게 되면 인지적 마찰이 발생하게 됩니다.
