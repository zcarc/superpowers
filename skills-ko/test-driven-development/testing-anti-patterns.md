# 테스팅 안티 패턴 (Testing Anti-Patterns)

**다음과 같은 경우 이 참고 문헌을 로드하십시오:** 테스트를 작성하거나 변경할 때, 모의 객체(mock)를 추가할 때, 또는 프로덕션 코드에 테스트 전용 메서드를 추가하고 싶은 유혹이 들 때.

## 개요

테스트는 모의 객체의 동작이 아니라 **실제 동작**을 검증해야 합니다. 모의 객체는 격리를 위한 수단일 뿐, 테스트의 대상이 아닙니다.

**핵심 원칙:** 모의 객체가 무엇을 하는지가 아니라, 코드가 무엇을 하는지를 테스트하십시오.

**엄격한 TDD를 따르면 이러한 안티 패턴을 방지할 수 있습니다.**

## 철칙 (The Iron Laws)

```
1. 절대로 모의 객체(mock)의 동작을 테스트하지 말 것
2. 절대로 프로덕션 클래스에 테스트 전용 메서드를 추가하지 말 것
3. 절대로 의존성을 이해하지 못한 채 모킹하지 말 것
```

## 안티 패턴 1: 모의 객체 동작 테스트

**위반 사례:**
```typescript
// ❌ 나쁜 예: 모의 객체가 존재하는지만 테스트함
test('사이드바를 렌더링한다', () => {
  render(<Page />);
  expect(screen.getByTestId('sidebar-mock')).toBeInTheDocument();
});
```

**왜 잘못되었는가:**
- 컴포넌트가 작동하는지가 아니라, 모의 객체가 작동하는지를 검증하고 있습니다.
- 모의 객체가 있으면 통과하고, 없으면 실패합니다.
- 실제 동작에 대해서는 아무것도 알려주지 않습니다.

**사용자의 교정:** "우리가 지금 모의 객체의 동작을 테스트하고 있는 건가요?"

**해결책:**
```typescript
// ✅ 좋은 예: 실제 컴포넌트를 테스트하거나 모킹하지 않음
test('사이드바를 렌더링한다', () => {
  render(<Page />);  // 사이드바를 모킹하지 않음
  expect(screen.getByRole('navigation')).toBeInTheDocument();
});

// 또는 격리를 위해 사이드바를 반드시 모킹해야 하는 경우:
// 모의 객체 자체를 단언(assert)하지 말고, 사이드바가 있을 때 Page가 어떻게 동작하는지 테스트하십시오.
```

### 가드 함수 (Gate Function)

```
모의 객체 요소를 단언(assert)하기 전에:
  질문: "내가 지금 실제 컴포넌트의 동작을 테스트하고 있나, 아니면 단순히 모의 객체의 존재 여부를 테스트하고 있나?"

  만약 단순히 존재 여부를 테스트하고 있다면:
    중단 - 해당 단언을 삭제하거나 컴포넌트 모킹을 해제하십시오.

  대신 실제 동작을 테스트하십시오.
```

## 안티 패턴 2: 프로덕션의 테스트 전용 메서드

**위반 사례:**
```typescript
// ❌ 나쁜 예: 테스트에서만 사용되는 destroy() 메서드
class Session {
  async destroy() {  // 프로덕션 API처럼 보임!
    await this._workspaceManager?.destroyWorkspace(this.id);
    // ... 정리 작업
  }
}

// 테스트 코드에서
afterEach(() => session.destroy());
```

**왜 잘못되었는가:**
- 프로덕션 클래스가 테스트 전용 코드로 오염됩니다.
- 프로덕션에서 실수로 호출될 경우 위험할 수 있습니다.
- YAGNI 원칙과 관심사 분리 원칙을 위반합니다.
- 객체의 생명주기와 엔티티의 생명주기를 혼동하게 만듭니다.

**해결책:**
```typescript
// ✅ 좋은 예: 테스트 유틸리티에서 테스트 정리 작업을 처리함
// Session에는 destroy()가 없음 - 프로덕션에서는 상태가 유지되지 않음(stateless)

// test-utils/ 파일에서
export async function cleanupSession(session: Session) {
  const workspace = session.getWorkspaceInfo();
  if (workspace) {
    await workspaceManager.destroyWorkspace(workspace.id);
  }
}

// 테스트 코드에서
afterEach(() => cleanupSession(session));
```

### 가드 함수 (Gate Function)

```
프로덕션 클래스에 메서드를 추가하기 전에:
  질문: "이게 테스트에서만 사용되는가?"

  만약 그렇다면:
    중단 - 추가하지 마십시오.
    대신 테스트 유틸리티에 넣으십시오.

  질문: "이 클래스가 해당 리소스의 생명주기를 소유하고 있는가?"

  만약 아니라면:
    중단 - 이 메서드가 있을 올바른 클래스가 아닙니다.
```

## 안티 패턴 3: 이해 없는 모킹

**위반 사례:**
```typescript
// ❌ 나쁜 예: 모킹이 테스트 로직을 깨뜨림
test('중복 서버를 감지한다', () => {
  // 모킹 때문에 테스트가 의존하는 설정 쓰기 작업이 생략됨!
  vi.mock('ToolCatalog', () => ({
    discoverAndCacheTools: vi.fn().mockResolvedValue(undefined)
  }));

  await addServer(config);
  await addServer(config);  // 에러를 던져야 하지만, 던지지 않음!
});
```

**왜 잘못되었는가:**
- 모킹된 메서드가 테스트에 필요한 부수 효과(설정 파일 쓰기 등)를 가지고 있었습니다.
- "안전하게 하기 위해" 과도하게 모킹하면 실제 동작이 깨집니다.
- 잘못된 이유로 테스트가 통과하거나 원인 불명으로 실패하게 됩니다.

**해결책:**
```typescript
// ✅ 좋은 예: 올바른 수준에서 모킹
test('중복 서버를 감지한다', () => {
  // 느린 부분만 모킹하고, 테스트에 필요한 동작은 보존함
  vi.mock('MCPServerManager'); // 느린 서버 시작 부분만 모킹

  await addServer(config);  // 설정이 정상적으로 기록됨
  await addServer(config);  // 중복 감지 성공 ✓
});
```

### 가드 함수 (Gate Function)

```
어떤 메서드를 모킹하기 전에:
  중단 - 아직 모킹하지 마십시오.

  1. 질문: "실제 메서드는 어떤 부수 효과(side effects)를 가지고 있는가?"
  2. 질문: "이 테스트가 그 부수 효과 중 하나라도 의존하고 있는가?"
  3. 질문: "이 테스트에 무엇이 필요한지 완전히 이해하고 있는가?"

  만약 부수 효과에 의존한다면:
    더 낮은 수준(실제로 느리거나 외부와 통신하는 작업)에서 모킹하십시오.
    또는 필요한 동작을 보존하는 테스트 더블(test doubles)을 사용하십시오.
    테스트가 의존하는 고수준 메서드를 통째로 모킹하지 마십시오.

  테스트가 무엇에 의존하는지 확신할 수 없다면:
    먼저 실제 구현체로 테스트를 실행해 보십시오.
    실제로 어떤 일이 일어나야 하는지 관찰하십시오.
    그 다음 필요한 수준에서 최소한의 모킹을 추가하십시오.

  주의 신호:
    - "안전하게 하려고 모킹할게요"
    - "이건 느릴 것 같으니 모킹하는 게 좋겠어요"
    - 의존성 체인을 이해하지 못한 채 모킹하는 경우
```

## 안티 패턴 4: 불완전한 모킹

**위반 사례:**
```typescript
// ❌ 나쁜 예: 부분적 모킹 - 필요하다고 생각하는 필드만 넣음
const mockResponse = {
  status: 'success',
  data: { userId: '123', name: 'Alice' }
  // 누락됨: 하위 코드에서 사용하는 메타데이터
};

// 나중에 response.metadata.requestId에 접근할 때 코드가 깨짐
```

**왜 잘못되었는가:**
- **부분적 모킹은 구조적 가정을 가립니다** - 여러분이 알고 있는 필드만 모킹하게 됩니다.
- **하위 코드가 여러분이 포함하지 않은 필드에 의존할 수 있습니다** - 조용한 실패(Silent failures)가 발생합니다.
- **테스트는 통과하지만 통합 테스트는 실패합니다** - 모킹은 불완전하고 실제 API는 완전하기 때문입니다.
- **잘못된 신뢰** - 테스트가 실제 동작에 대해 아무것도 증명하지 못합니다.

**철칙:** 현재 테스트에서 사용하는 필드뿐만 아니라, 실제 존재하는 **완전한** 데이터 구조를 모킹하십시오.

**해결책:**
```typescript
// ✅ 좋은 예: 실제 API의 완전함을 반영함
const mockResponse = {
  status: 'success',
  data: { userId: '123', name: 'Alice' },
  metadata: { requestId: 'req-789', timestamp: 1234567890 }
  // 실제 API가 반환하는 모든 필드 포함
};
```

### 가드 함수 (Gate Function)

```
모의 응답(mock response)을 만들기 전에:
  확인: "실제 API 응답에는 어떤 필드들이 포함되어 있는가?"

  실행:
    1. 문서나 예시를 통해 실제 API 응답을 조사하십시오.
    2. 시스템 하위에서 소비될 수 있는 모든 필드를 포함하십시오.
    3. 모의 객체가 실제 응답 스키마와 완전히 일치하는지 확인하십시오.

  중요:
    모의 객체를 만든다면 전체 구조를 반드시 이해해야 합니다.
    부분적 모킹은 코드가 생략된 필드에 의존할 때 조용히 실패합니다.

  불확실한 경우: 문서화된 모든 필드를 포함하십시오.
```

## 안티 패턴 5: 사후 테스트

**위반 사례:**
```
✅ 구현 완료
❌ 테스트 작성 안 됨
"테스트할 준비가 되었습니다"
```

**왜 잘못되었는가:**
- 테스팅은 구현의 일부이지, 선택적인 후속 작업이 아닙니다.
- TDD를 따랐다면 이런 상황이 발생하지 않았을 것입니다.
- 테스트 없이는 완료되었다고 주장할 수 없습니다.

**해결책:**
```
TDD 사이클:
1. 실패하는 테스트 작성
2. 통과하도록 구현
3. 리팩토링
4. 그 다음에 완료 선언
```

## 모의 객체가 너무 복잡해질 때

**경고 신호:**
- 모의 객체 설정이 테스트 로직보다 깁니다.
- 테스트를 통과시키기 위해 모든 것을 모킹하고 있습니다.
- 실제 컴포넌트가 가진 메서드가 모의 객체에는 누락되어 있습니다.
- 모의 객체가 바뀌면 테스트가 깨집니다.

**사용자의 질문:** "여기서 반드시 모의 객체를 사용해야 하나요?"

**고려사항:** 실제 컴포넌트를 사용하는 통합 테스트가 복잡한 모킹보다 대개 더 단순합니다.

## TDD는 이러한 안티 패턴을 방지합니다

**TDD가 도움이 되는 이유:**
1. **테스트 먼저 작성** → 내가 실제로 무엇을 테스트하는지 생각하게 강제합니다.
2. **실패 지켜보기** → 테스트가 모의 객체가 아닌 실제 동작을 검증하는지 확인합니다.
3. **최소 구현** → 테스트 전용 메서드가 끼어들 틈이 없습니다.
4. **실제 의존성** → 모킹하기 전에 테스트에 실제로 무엇이 필요한지 알게 됩니다.

**만약 모의 객체의 동작을 테스트하고 있다면 TDD를 위반한 것입니다.** 실제 코드를 대상으로 실패하는 모습을 먼저 보지 않은 채 모의 객체를 추가했기 때문입니다.

## 빠른 참조

| 안티 패턴 | 해결책 |
|--------------|-----|
| 모의 객체 요소를 단언함 | 실제 컴포넌트를 테스트하거나 모킹을 해제함 |
| 프로덕션에 테스트 전용 메서드 추가 | 테스트 유틸리티로 이동함 |
| 이해 없는 모킹 | 의존성을 먼저 이해하고 최소한으로 모킹함 |
| 불완전한 모킹 | 실제 API를 완전히 반영함 |
| 사후 테스트 | TDD - 테스트를 먼저 작성함 |
| 과도하게 복잡한 모킹 | 통합 테스트 사용을 고려함 |

## 주의 신호 (Red Flags)

- `*-mock` 테스트 ID를 확인하는 단언문
- 테스트 파일에서만 호출되는 메서드
- 모의 객체 설정이 테스트의 50% 이상을 차지함
- 모의 객체를 제거하면 테스트가 실패함
- 모의 객체가 왜 필요한지 설명하지 못함
- "안전하게 하려고" 모킹하는 경우

## 결론

**모의 객체는 격리를 위한 도구이지, 테스트 대상이 아닙니다.**

TDD 과정에서 여러분이 모의 객체의 동작을 테스트하고 있다는 사실을 깨달았다면 무언가 잘못된 것입니다.

해결책: 실제 동작을 테스트하거나, 왜 모킹을 하고 있는지 근본적으로 검토하십시오.
