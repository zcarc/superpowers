# 계획 실행 (Executing Plans)

## 개요

계획을 로드하고, 비판적으로 검토하며, 모든 작업을 실행하고, 완료 시 보고합니다.

**시작 시 공지:** "이 계획을 구현하기 위해 executing-plans 기술을 사용하고 있습니다."

**참고:** 사용자에게 Superpowers는 서브에이전트 접근 권한이 있을 때 훨씬 더 잘 작동한다고 알리십시오. 서브에이전트 지원이 있는 플랫폼에서 실행될 때 작업 품질이 훨씬 높아집니다. 서브에이전트를 사용할 수 있다면 이 기술 대신 `superpowers:subagent-driven-development`를 사용하십시오.

## 프로세스

### 1단계: 계획 로드 및 검토
1. 계획 파일 읽기
2. 비판적으로 검토 - 계획에 대한 질문이나 우려 사항 식별
3. 우려 사항이 있는 경우: 시작하기 전에 사용자에게 제기
4. 우려 사항이 없는 경우: Todo 작성 후 진행

### 2단계: 작업 실행

각 작업에 대해:
1. `in_progress`로 표시
2. 각 단계를 정확히 따름 (계획에는 한입 크기의 단계들이 있음)
3. 명시된 대로 검증 실행
4. 완료로 표시

### 3단계: 개발 완료

모든 작업이 완료되고 검증된 후:
- 공지: "이 작업을 완료하기 위해 finishing-a-development-branch 기술을 사용하고 있습니다."
- **필수 하위 기술:** `superpowers:finishing-a-development-branch` 사용
- 해당 기술의 안내에 따라 테스트를 확인하고, 옵션을 제시하며, 선택된 내용을 실행

## 중단 및 도움 요청 시기

**다음과 같은 경우 즉시 실행을 중단하십시오:**
- 차단 요소 발생 (누락된 의존성, 테스트 실패, 지침 불분명)
- 시작을 방해하는 계획의 중대한 결함 발견
- 지침을 이해하지 못함
- 검증이 반복적으로 실패함

**추측하기보다 명확한 설명을 요청하십시오.**

## 이전 단계로 돌아가야 할 때

**다음과 같은 경우 검토(1단계)로 돌아가십시오:**
- 사용자가 여러분의 피드백을 바탕으로 계획을 업데이트함
- 근본적인 접근 방식의 재고가 필요함

**차단 요소를 억지로 통과하려 하지 마십시오** - 중단하고 질문하십시오.

## 기억할 것
- 먼저 계획을 비판적으로 검토하십시오.
- 계획의 단계를 정확히 따르십시오.
- 검증을 건너뛰지 마십시오.
- 계획에서 언급할 때 해당 기술을 참조하십시오.
- 막히면 중단하고 추측하지 마십시오.
- 사용자의 명시적인 동의 없이 main/master 브랜치에서 구현을 시작하지 마십시오.

## 통합

**필수 워크플로우 기술:**
- **superpowers:using-git-worktrees** - 필수: 시작 전 격리된 작업 공간 설정
- **superpowers:writing-plans** - 이 기술이 실행할 계획을 생성
- **superpowers:finishing-a-development-branch** - 모든 작업 후 개발 완료
