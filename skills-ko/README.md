# skills-ko 메모

이 문서는 `skills-ko/receiving-code-review/SKILL.md`와 `skills-ko/subagent-driven-development/` 관련 프롬프트가 이 저장소에서 언제, 어디서, 어떻게 호출되는지 정리한 메모입니다.

## 결론

`receiving-code-review`는 다른 스킬이나 스크립트가 자동으로 이어서 호출하는 하위 스킬이 아닙니다. 리뷰 피드백을 받은 상황에서 에이전트가 이름으로 직접 로드하는 독립 스킬입니다.

## 언제 호출되는가

`receiving-code-review`는 코드 리뷰 피드백을 받은 직후, 그리고 제안 사항을 구현하기 전에 사용하는 스킬입니다.

특히 다음 상황을 대상으로 합니다.

- 피드백이 모호해서 바로 구현하면 위험할 때
- 외부 리뷰어 제안이 기술적으로 맞는지 검증이 필요할 때
- 기존 동작, 호환성, YAGNI 관점에서 반론 가능성이 있을 때

핵심 동작은 다음 순서입니다.

1. 피드백 전체 읽기
2. 요구 사항 재진술 또는 질문
3. 코드베이스 기준으로 검증
4. 기술적으로 적합한지 평가
5. 수용 또는 반론
6. 항목별 구현 및 테스트

## 어디서 호출되는가

이 저장소의 스킬 로딩 구조상, 세션 시작 시 `hooks/session-start`가 `using-superpowers`를 주입합니다.

`using-superpowers`는 다음 원칙을 줍니다.

- 관련 있거나 요청된 스킬은 응답이나 조치 전에 호출해야 함
- 스킬은 파일을 직접 읽는 대신 스킬 도구로 로드해야 함

즉 `receiving-code-review`는 별도 코드 경로에서 자동 실행되는 것이 아니라, "지금은 리뷰 피드백에 대응하는 상황이다"라고 판단될 때 스킬 시스템을 통해 선택되는 구조입니다.

README에서도 이 스킬은 Collaboration 카테고리의 독립 항목으로만 나옵니다.

- `requesting-code-review` - 리뷰 요청 전 체크리스트
- `receiving-code-review` - 리뷰 피드백에 대한 대응

## 어떻게 호출되는가

호출 방식은 명시적 스킬 호출입니다. 즉 `superpowers:receiving-code-review` 같은 이름으로 직접 로드되는 형태입니다.

현재 저장소 검색 기준으로는 다음 특징이 확인됩니다.

- `subagent-driven-development`는 통합 섹션에서 `superpowers:requesting-code-review`를 명시적으로 참조함
- `receiving-code-review`에 대해서는 같은 형태의 통합 참조가 없음
- 테스트 목록에도 `requesting-code-review` 트리거 테스트는 있으나 `receiving-code-review` 전용 트리거 테스트는 보이지 않음

따라서 `receiving-code-review`는 워크플로우에 자동 연결된 스킬이라기보다, 리뷰 코멘트를 받은 후 필요 시 직접 선택해서 쓰는 스킬로 보는 것이 맞습니다.

## 관련 스킬과의 차이

### requesting-code-review

리뷰를 받기 전에 사용하는 스킬입니다.

- 구현 완료 후
- 주요 기능 완료 후
- `main` 머지 전

리뷰어 서브에이전트를 파견하고, SHA와 요구 사항을 정리해 전달하는 방법을 설명합니다.

### receiving-code-review

리뷰를 받은 뒤 사용하는 스킬입니다.

- 피드백을 그대로 따르지 않고 먼저 검증함
- 필요하면 질문하거나 반론함
- 맞는 항목만 순서대로 구현하고 각각 테스트함

## 한 줄 요약

`requesting-code-review`는 리뷰를 요청할 때, `receiving-code-review`는 리뷰 결과를 처리할 때 쓰며, 후자는 자동 호출이 아니라 상황에 맞춘 직접 호출용 독립 스킬입니다.

## subagent-driven-development 프롬프트 호출 분석

이 섹션은 다음 세 파일이 언제, 어디서, 어떻게, 왜 호출되는지 정리합니다.

- `skills-ko/subagent-driven-development/spec-reviewer-prompt.md`
- `skills-ko/subagent-driven-development/implementer-prompt.md`
- `skills-ko/subagent-driven-development/code-quality-reviewer-prompt.md`

### 결론

이 세 파일은 일반 코드에서 함수처럼 직접 호출되는 파일이 아닙니다. `subagent-driven-development/SKILL.md`에 정의된 워크플로우 안에서, 컨트롤러 에이전트가 서브에이전트를 파견할 때 사용하는 프롬프트 템플릿입니다.

즉 실행 구조는 다음과 같습니다.

1. 구현 서브에이전트 파견
2. 사양 준수 리뷰 서브에이전트 파견
3. 코드 품질 리뷰 서브에이전트 파견
4. 이슈가 있으면 같은 구현자가 수정하고 다시 리뷰

핵심은 `implementer -> spec reviewer -> code quality reviewer` 순서이며, 두 리뷰는 모두 루프 구조입니다.

### 언제 호출되는가

#### 1. `implementer-prompt.md`

각 작업의 시작점에서 호출됩니다.

- 계획 파일을 한 번 읽고 작업들을 모두 추출한 뒤 첫 작업을 시작할 때
- 구현자가 질문해서 추가 문맥이 필요할 때 답변 후 재파견할 때
- 이전 작업이 끝나고 다음 작업으로 넘어갈 때
- 리뷰어가 이슈를 찾았고 같은 구현자가 수정 작업을 이어서 할 때

이 템플릿은 작업별 구현 담당 서브에이전트를 띄우는 시작점입니다.

#### 2. `spec-reviewer-prompt.md`

구현자가 작업을 완료했다고 보고한 직후 호출됩니다.

- 구현 결과가 요구사항과 정확히 일치하는지 확인할 때
- 빠진 요구사항, 과구현, 요구사항 오해를 검증할 때
- 이슈가 발견되면 구현자가 수정한 뒤 재리뷰할 때

즉 코드 품질 리뷰보다 먼저 실행되는 첫 번째 품질 게이트입니다.

#### 3. `code-quality-reviewer-prompt.md`

사양 준수 리뷰가 통과한 뒤에만 호출됩니다.

- 구현이 요구사항에 맞다는 것이 확인된 뒤
- 코드 품질, 테스트, 유지보수성, 파일 책임 분리를 점검할 때
- 품질 이슈가 있으면 구현자가 수정한 뒤 재리뷰할 때

즉 두 번째 품질 게이트이며, spec review 통과 전에는 시작하면 안 됩니다.

### 어디서 호출되는가

호출 주체는 모두 `subagent-driven-development/SKILL.md`입니다. 이 스킬은 구현 계획이 있고, 작업들이 비교적 독립적이며, 현재 세션 안에서 계속 실행할 때 사용하는 워크플로우입니다.

저장소 관점에서 보면:

- 이 세 파일은 스킬 문서에서 프롬프트 템플릿으로 참조됩니다
- 테스트와 릴리즈 노트에서도 같은 순서와 역할로 설명됩니다
- 일반 애플리케이션 코드에서 import 하거나 실행하는 별도 런타임 코드는 보이지 않습니다

따라서 "어디서 호출되느냐"에 대한 정확한 답은, 애플리케이션 코드 경로가 아니라 `subagent-driven-development` 스킬의 컨트롤러 절차 안에서 호출된다는 것입니다.

### 어떻게 호출되는가

#### `implementer-prompt.md`

범용 Task 도구 프롬프트로 호출됩니다.

- 작업 이름
- 계획에서 가져온 작업 전체 텍스트
- 이 작업이 어디에 들어가는지 설명하는 문맥
- 작업 디렉터리

를 직접 프롬프트에 넣어서 서브에이전트를 파견합니다.

중요한 점:

- 서브에이전트에게 계획 파일을 직접 읽게 하지 않습니다
- 컨트롤러가 필요한 작업 텍스트를 그대로 붙여 넣습니다
- 구현자는 질문 가능하며, 구현, 테스트, 커밋, self-review 후 상태를 보고합니다
- 상태는 `DONE`, `DONE_WITH_CONCERNS`, `BLOCKED`, `NEEDS_CONTEXT` 중 하나입니다

#### `spec-reviewer-prompt.md`

이것도 범용 Task 도구 프롬프트로 호출됩니다.

프롬프트에는 다음이 들어갑니다.

- 요구사항 전체 텍스트
- 구현자가 자신이 무엇을 만들었다고 보고한 내용

하지만 이 리뷰어는 구현자의 보고를 믿지 말고 실제 코드를 직접 읽으라는 강한 지시를 받습니다.

리뷰 포인트:

- 빠진 요구사항이 있는가
- 요청되지 않은 기능을 더 만들었는가
- 요구사항을 잘못 해석했는가

출력은 크게 두 가지입니다.

- `✅ Spec compliant`
- `❌ Issues found: ...`

즉 이 단계의 목적은 "잘 만든 코드인가"보다 먼저 "요구한 것을 정확히 만들었는가"를 확인하는 것입니다.

#### `code-quality-reviewer-prompt.md`

이 파일은 직접 리뷰 지침을 모두 담고 있기보다, `superpowers:code-reviewer`를 다시 호출하는 래퍼 역할을 합니다.

프롬프트에는 다음 정보가 들어갑니다.

- 구현자가 무엇을 구현했다고 보고했는지
- 어떤 작업 요구사항을 기준으로 볼 것인지
- `BASE_SHA`
- `HEAD_SHA`
- 작업 설명

그리고 실제 리뷰 템플릿은 `requesting-code-review/code-reviewer.md`를 사용합니다. 즉 이 단계는 git diff 범위를 기준으로 코드 리뷰를 수행하는 구조입니다.

추가로 보는 항목은 다음과 같습니다.

- 각 파일이 하나의 명확한 책임을 가지는가
- 독립적으로 이해하고 테스트할 수 있게 나뉘어 있는가
- 계획에서 정한 파일 구조를 따르는가
- 이번 변경이 새 파일이나 기존 파일을 과도하게 비대하게 만들었는가

### 왜 이렇게 호출되는가

이 구조의 이유는 역할 분리와 품질 게이트를 명확히 하기 위해서입니다.

- 구현자는 작업을 완수하고 self-review까지 수행합니다
- spec reviewer는 요구사항 일치 여부만 회의적으로 검증합니다
- code quality reviewer는 그 다음에야 코드 품질과 유지보수성을 봅니다

이 순서를 강제하는 이유는 "코드는 깔끔하지만 요구사항이 틀린" 실패를 막기 위해서입니다. 먼저 사양 적합성을 확인하고, 그 다음에 품질을 점검하는 편이 더 정확합니다.

또한 컨트롤러가 매번 필요한 텍스트와 문맥만 제공하므로:

- 서브에이전트 컨텍스트 오염이 줄고
- 계획 파일 재독 비용이 줄며
- 작업별 집중도가 높아지고
- 리뷰도 구현자 보고가 아니라 실제 결과물 중심으로 진행됩니다

### 한 줄 요약

`implementer-prompt.md`는 작업 구현용, `spec-reviewer-prompt.md`는 요구사항 일치 검증용, `code-quality-reviewer-prompt.md`는 사양 통과 후 코드 품질 검증용이며, 모두 `subagent-driven-development` 스킬의 작업별 루프 안에서 순서대로 호출됩니다.
