# 완료 전 검증 (Verification Before Completion)

## 개요

검증 없이 작업을 완료했다고 주장하는 것은 효율적인 것이 아니라 정직하지 못한 것입니다.

**핵심 원칙:** 주장하기 전에 항상 증거를 제시하십시오.

**이 규칙의 문구(letter)를 어기는 것은 규칙의 본질(spirit)을 어기는 것과 같습니다.**

## 철칙 (The Iron Law)

```
최신 검증 증거 없이는 완료를 주장하지 말 것
```

이 메시지에서 검증 명령을 직접 실행하지 않았다면, 통과했다고 주장할 수 없습니다.

## 가드 함수 (Gate Function)

```
상태를 주장하거나 만족감을 표현하기 전에:

1. 식별: 이 주장을 증명할 수 있는 명령은 무엇인가?
2. 실행: 전체 명령을 실행 (새롭고 완전하게)
3. 확인: 전체 출력을 읽고, 종료 코드(exit code)를 확인하며, 실패 횟수를 계산
4. 검증: 출력이 주장을 뒷받침하는가?
   - 아니오: 증거와 함께 실제 상태를 기술함
   - 예: 증거와 함께 주장을 기술함
5. 그 다음에만: 완료를 주장함

단계를 하나라도 건너뛰는 것 = 검증이 아니라 거짓말을 하는 것
```

## 흔한 실패 사례

| 주장 | 필요한 것 | 불충분한 것 |
|-------|----------|----------------|
| 테스트 통과 | 테스트 명령 출력: 실패 0건 | 이전 실행 결과, "통과할 것임" |
| 린터 통과 | 린터 출력: 에러 0건 | 부분적인 확인, 추정 |
| 빌드 성공 | 빌드 명령: 종료 코드 0 | 린터 통과, 로그가 좋아 보임 |
| 버그 수정 | 원래의 증상 테스트: 통과 | 코드 변경, 수정되었다고 가정 |
| 회귀 테스트 작동 | 레드-그린 사이클 검증됨 | 테스트가 한 번 통과함 |
| 에이전트 완료 | VCS diff가 변경 사항을 보여줌 | 에이전트가 "성공"이라고 보고함 |
| 요구 사항 충족 | 한 줄씩 체크리스트 확인 | 테스트 통과 |

## 주의 신호 (Red Flags) - 즉시 중단하십시오

- "~일 것이다", "아마도", "~인 것 같다" 등의 표현 사용
- 검증 전에 만족감 표시 ("좋아요!", "완벽해요!", "완료!", 등)
- 검증 없이 커밋/푸시/PR 하려는 경우
- 에이전트의 성공 보고를 그대로 믿는 경우
- 부분적인 검증에 의존하는 경우
- "이번 한 번만"이라고 생각하는 경우
- 피곤해서 작업을 끝내고 싶어 하는 경우
- **검증 명령을 실행하지 않은 상태에서 성공을 암시하는 모든 단어 사용**

## 합리화 방지

| 변명 | 현실 |
|--------|---------|
| "이제 작동할 거예요" | 검증 명령을 실행하십시오. |
| "확신합니다" | 확신은 증거가 아닙니다. |
| "이번 한 번만요" | 예외는 없습니다. |
| "린터가 통과했으니까요" | 린터 통과가 컴파일 성공을 의미하지 않습니다. |
| "에이전트가 성공했대요" | 독립적으로 검증하십시오. |
| "피곤해요" | 피로는 변명이 될 수 없습니다. |
| "부분적 확인으로 충분해요" | 부분적인 것은 아무것도 증명하지 못합니다. |
| "다른 단어를 썼으니 규칙 위반이 아니죠" | 문구가 아니라 본질이 중요합니다. |

## 주요 패턴

**테스트:**
```
✅ [테스트 명령 실행] [확인: 34/34 통과] "모든 테스트가 통과했습니다"
❌ "이제 통과할 거예요" / "정확해 보이네요"
```

**회귀 테스트 (TDD 레드-그린):**
```
✅ 작성 → 실행 (통과) → 수정사항 되돌리기 → 실행 (반드시 실패해야 함) → 복구 → 실행 (통과)
❌ "회귀 테스트를 작성했습니다" (레드-그린 검증 없이)
```

**빌드:**
```
✅ [빌드 실행] [확인: 종료 코드 0] "빌드 성공"
❌ "린터 통과" (린터는 컴파일 가능 여부를 체크하지 않음)
```

**요구 사항:**
```
✅ 계획 다시 읽기 → 체크리스트 생성 → 각각 검증 → 누락 또는 완료 보고
❌ "테스트 통과, 단계 완료"
```

**에이전트 위임:**
```
✅ 에이전트가 성공 보고 → VCS diff 확인 → 변경 사항 검증 → 실제 상태 보고
❌ 에이전트 보고를 그대로 믿음
```

## 이것이 중요한 이유

과거의 실패 사례들로부터:
- 사용자가 "당신을 믿지 않습니다"라고 말함 - 신뢰가 깨짐
- 정의되지 않은 함수가 포함된 채 배포됨 - 크래시 발생
- 누락된 요구 사항이 포함된 채 배포됨 - 불완전한 기능
- 잘못된 완료 보고로 인한 시간 낭비 → 방향 수정 → 재작업
- "정직함은 핵심 가치입니다. 거짓말을 하면 교체될 것입니다." 원칙 위반

## 적용 시점

**항상 다음의 경우 이전에 적용하십시오:**
- 성공/완료 주장의 모든 변형
- 만족감의 표현
- 작업 상태에 대한 긍정적인 진술
- 커밋, PR 생성, 작업 완료 선언
- 다음 작업으로 이동
- 에이전트에게 위임

**다음의 경우에 모두 적용됩니다:**
- 정확한 문구
- 의역 및 동의어
- 성공의 암시
- 완료/정확성을 시사하는 모든 의사소통

## 결론

**검증에는 지름길이 없습니다.**

명령을 실행하십시오. 출력을 읽으십시오. **그 다음에만** 결과를 주장하십시오.

이것은 협상의 대상이 아닙니다.
