# 스킬 및 에이전트 흐름 검토 (Skill and Agent Flow Review)

날짜: 2026-04-01
프로젝트: `timer`
범위: 이 문서는 현재 세션에서 실제로 일어난 일, 발생 이유, 개인 프로젝트의 빠르고 안전한 워크플로우에 비해 과도했던 부분, 그리고 스킬 및 에이전트 트리거 규칙 개선 방안을 검토합니다.

## 요약 (Executive Summary)

현재 Superpowers 설정은 빠른 반복이 필요한 개인 프로젝트보다는 규율이 강조되는 팀 워크플로우에 더 최적화되어 있습니다.

이번 세션에서의 실제 흐름은 다음과 같았습니다:

`using-superpowers -> systematic-debugging -> brainstorming -> writing-plans -> executing-plans -> test-driven-development -> implementation -> requesting-code-review -> review subagent(s) -> receiving-code-review -> re-review subagent(s) -> finishing-a-development-branch -> verification-before-completion`

이는 단순한 버그 수정을 하기에는 필요 이상으로 무거웠습니다.

가장 큰 불일치 지점은 다음과 같습니다:

1. `인라인 실행(Inline Execution)`을 선택했음에도 불구하고 리뷰 서브 에이전트가 파견되었습니다.
2. 리뷰 서브 에이전트 자체가 추가적인 스킬을 호출했다고 보고했습니다.
3. 작은 버그 수정 작업임에도 설계 승인, 계획 작성, 실행 워크플로우, 반복적인 리뷰 및 브랜치 종료 흐름을 모두 거쳤습니다.

속도가 중요하면서도 정확성 또한 놓칠 수 없는 개인 프로젝트의 경우, 더 나은 기본값은 다음과 같습니다:

`using-superpowers -> systematic-debugging -> test-driven-development -> 인라인 구현(inline implementation) -> 완료 전 검증(verification-before-completion)`

그 후 변경 사항의 규모나 위험도가 정당화될 때만 계획 수립이나 리뷰 레이어를 추가하는 것이 바람직합니다.

## 실제 세션 타임라인 (Actual Session Timeline)

이 섹션은 이번 세션에서 실제로 일어난 일을 순서대로 기록합니다.

### 1. 메인 에이전트 시작

트리거된 컴포넌트:
- `using-superpowers`

실행 이유:
- 현재 루트 규칙에 따라 응답이나 행동 전에 관련 스킬을 호출하도록 되어 있습니다.
- `skills-custom/using-superpowers/SKILL.md`는 대화 시작 시 이를 필수로 규정합니다.

평가:
- 루트 부트스트랩 스킬로서 적절합니다.

### 2. 초기 버그 조사

트리거된 컴포넌트:
- `systematic-debugging`

실행 이유:
- 사용자가 잘못된 동작을 보고했습니다: 커스텀 5초 작업 항목이 리뷰 화면에서 `예정 1분`으로 표시되는 문제.
- 이는 버그 보고이며 디버깅 워크플로우에 적합합니다.

평가:
- 적절합니다.

### 3. 디자인 모드 전환

트리거된 컴포넌트:
- `brainstorming`

실행 이유:
- 사용자가 `설계 진행`이라고 말했습니다.
- 현재 브레인스토밍 스킬은 모든 수정 작업을 설계 및 승인 단계 뒤에 배치합니다.

일어난 일:
- 프로젝트 컨텍스트를 탐색했습니다.
- 범위(scope)에 대한 질문을 던졌습니다.
- 3가지 접근 방식을 제안했습니다.
- 권장 디자인을 제시했습니다.
- 사용자가 디자인을 승인했습니다.

평가:
- 스킬 정의상 기술적으로는 일관성이 있으나, 작고 범위가 명확한 버그 수정에는 무거웠습니다.

### 4. 계획 생성

트리거된 컴포넌트:
- `writing-plans`

실행 이유:
- `brainstorming` 스킬은 디자인 승인 후 다음 단계로 `writing-plans`를 명시하고 있습니다.

일어난 일:
- 상세 계획 파일이 다음 경로에 작성되었습니다:
  - `docs/superpowers/plans/2026-04-01-work-mode-subminute-duration-display-plan.md`
- 계획에서 다음을 권장했습니다:
  - `1. 인라인 실행(Inline Execution)`

평가:
- 현재 스킬 체인상 일관성이 있으나, 개인 프로젝트 맥락에서 이 수정을 위해 하기에는 과했습니다.

### 5. 인라인 실행 선택

트리거된 컴포넌트:
- `executing-plans`

실행 이유:
- 계획에 `인라인 실행을 선택할 경우: 필수 서브 스킬: Use superpowers:executing-plans`라고 되어 있었습니다.
- 사용자가 `1`을 선택했습니다.

중요한 명확화:
- 현재 디자인에서 `인라인 실행`은 메인 에이전트가 구현 작업을 직접 수행한다는 의미입니다.
- 이것이 `서브 에이전트를 전혀 사용하지 않음`을 의미하지는 않습니다.

평가:
- 이 부분이 혼란을 야기한 핵심적인 명칭 불일치 지점이었습니다.

### 6. 테스트 우선 구현 (TDD)

트리거된 컴포넌트:
- `test-driven-development`

실행 이유:
- 현재 TDD 스킬은 기능 작업, 버그 수정, 리팩토링 및 동작 변경 시 사용해야 한다고 되어 있습니다.

일어난 일:
- 실패하는 테스트를 먼저 추가했습니다.
- 실패를 확인했습니다.
- 최소한의 구현을 추가했습니다.
- 테스트를 다시 실행했습니다.

평가:
- 적절하고 가치 있었습니다.

### 7. 메인 에이전트 구현 작업

트리거된 컴포넌트:
- 구현을 위한 별도 서브 에이전트 없음

일어난 일:
- 메인 에이전트가 다음 영역을 직접 수정하고 검증했습니다:

  - `src/renderer/lib/work-mode.ts`
  - `src/renderer/components/work-session-review.tsx`

  - 관련 테스트들

평가:
- 이 부분은 인라인 구현에 대한 사용자의 기대와 일치했습니다.

### 8. 구현 후 최종 리뷰 단계

트리거된 컴포넌트:
- `requesting-code-review`

실행 이유:
- `executing-plans`의 3단계는 작업 완료 후 최종 리뷰를 요구합니다.
- `requesting-code-review`는 코드 리뷰어 서브 에이전트를 파견하도록 명시하고 있습니다.

일어난 일:
- `code-reviewer` 파견 시도가 `ProviderModelNotFoundError`로 인해 첫 번째 실패했습니다.
- 이후 `general` 서브 에이전트가 리뷰어로 대신 사용되었습니다.

평가:
- 이 지점이 `인라인`임에도 서브 에이전트 사용으로 이어진 부분입니다.
- 현재 스킬 체인과는 일관되지만, `메인 에이전트 전용 인라인`을 기대했을 사용자에게는 불일치하는 동작이었습니다.

### 9. 메인 에이전트의 리뷰 결과 수신

트리거된 컴포넌트:
- `receiving-code-review`

실행 이유:
- 서브 에이전트로부터 리뷰 피드백이 도착했습니다.
- `receiving-code-review` 워크플로우는 메인 에이전트가 리뷰 피드백을 반영하기 전에 이를 검증하도록 설계되었습니다.

일어난 일:
- 메인 에이전트가 발견 사항을 확인했습니다.
- 리뷰 내용 중 하나를 수용하여 수정했습니다.
- 새로운 회귀 테스트를 추가했습니다.

평가:
- 일단 리뷰 서브 에이전트가 사용된 이상 합리적인 절차입니다.
- 하지만 이 레이어는 애초에 리뷰 서브 에이전트가 사용되었기 때문에 존재하는 것입니다.

### 10. 재리뷰 루프

일어난 일:
- 여러 번의 리뷰 서브 에이전트 세션이 사용되었습니다.

평가:
- 위험도가 높은 작업에는 유용합니다.
- 개인 프로젝트 워크플로우에서의 작은 UI 버그 수정치고는 너무 무거웠습니다.

### 11. 브랜치 종료 워크플로우

트리거된 컴포넌트:
- `finishing-a-development-branch`

실행 이유:

## 서브 에이전트 복귀 후 메인 에이전트의 동작 (Main Agent After Subagent Return)

여기서의 질문은: 서브 에이전트가 작업을 메인 에이전트에게 돌려주었을 때, 메인 에이전트가 다시 새로운 스킬이나 에이전트를 트리거했는가? 입니다.

네, 그런 일이 일어났습니다.

이번 세션의 실제 패턴:

1. 메인 에이전트가 리뷰 서브 에이전트를 파견함.
2. 리뷰 서브 에이전트가 결과를 반환함.
3. 메인 에이전트가 해당 결과를 평가하기 위해 `receiving-code-review`를 호출함.
4. 메인 에이전트가 수정을 가하고 테스트를 실행함.
5. 메인 에이전트가 확인을 위해 또 다른 리뷰 서브 에이전트를 파견함.
6. 최종 리뷰 후, 메인 에이전트가 `finishing-a-development-branch`와 `verification-before-completion`을 호출함.

즉, 현재 시스템은 다음을 허용합니다:

`메인 에이전트 -> 서브 에이전트 -> 메인 에이전트 스킬 -> 서브 에이전트 다시 -> 메인 에이전트 스킬`

이번 세션에서 실제로 이 일이 일어났습니다.

일어나지 않은 일:

- 리뷰 서브 에이전트가 추가적인 서브 에이전트를 파견하지 않았습니다.
- 리뷰 서브 에이전트가 `code-reviewer` 에이전트를 다시 파견하지 않았습니다.
- 무한 에이전트 재귀(recursion)는 없었습니다.

## 관찰된 핵심 문제점 (Core Problems Observed)

### 문제 1: `인라인 실행(Inline Execution)`의 오해 소지

현재의 실질적 의미:
- 메인 에이전트가 구현을 직접 수행함.
- 리뷰는 여전히 서브 에이전트를 사용할 수 있음.

사용자의 예상 기대치:
- 명시적으로 요청하지 않는 한 서브 에이전트 파견 없이 메인 에이전트가 처음부터 끝까지 작업을 수행함.

영향:
- 사용자 혼란.
- 워크플로우가 사용자가 선택한 모드를 위반하는 것으로 보임.

### 문제 2: 작은 수정 사항도 무거운 계획 단계를 거쳐야 함

현재 경로에 따르면 작은 버그 수정도 다음과 같이 될 수 있습니다:
- 디자인 -> 승인 -> 계획 문서 -> 실행 모드 선택 -> 실행 워크플로우 -> 코드 리뷰 워크플로우 -> 브랜치 종료 워크플로우.

영향:
- 반복 속도 저하.
- 토큰 사용량 증가.
- 코드 오류보다 워크플로우 이탈 가능성이 더 커짐.

### 문제 3: 리뷰 서브 에이전트가 충분히 격리되지 않음

리뷰 서브 에이전트들이 다음과 같은 스킬을 호출했다고 보고했습니다:
- `using-superpowers`
- `receiving-code-review`
- `verification-before-completion`

영향:
- 리뷰 작업자가 순수하게 리뷰 작업자로서만 동작하지 않음.
- 메인 에이전트의 영역인 워크플로우 로직에 재진입할 수 있음.
- 특히 리뷰 서브 에이전트 내부에서 `receiving-code-review`를 하는 것은 매우 어색함.

### 문제 4: 브랜치 종료 흐름이 너무 성급함

현재 이슈:
- 사용자가 커밋, 머지, 또는 PR 생성을 요청하지 않았음에도 시스템이 `finishing-a-development-branch`에 진입함.

영향:
- 불필요한 오버헤드.
- 관련 없는 명령어가 많아짐.
- 환경 특정적인 실패를 만날 가능성이 커짐.

### 문제 5: 검증 범위가 너무 넓음

현재 이슈:
- 전체 `npm test`에 관련 없는 `.opencode` 의존성 테스트가 포함됨.

영향:
- 노이즈 발생.
- 현재의 변경 사항이 프로젝트 외부의 이슈 때문에 가로막혀 있다는 착각을 일으킴.

## 스킬 및 에이전트별 권장 트리거 규칙 (Recommended Trigger Rules Per Skill And Agent)

이 섹션은 개인 프로젝트의 빠르고 안전한 워크플로우를 위해 더 적절한 트리거 조건을 권장합니다.

### 기본값으로 유지

#### `using-superpowers`

권장 트리거:
- 대화 시작 시에만.
- 리뷰 전용 서브 에이전트 내부에서는 재실행하지 않음.

#### `systematic-debugging`

권장 트리거:
- 버그 보고.
- 테스트 실패.
- 예상치 못한 동작.

#### `test-driven-development`

권장 트리거:
- 테스트 가능한 모든 동작 변경 또는 버그 수정.

#### `verification-before-completion`

권장 트리거:
- 완료를 주장하기 전.
- 수정을 유의미하게 증명할 수 있는 가장 좁은 범위의 명령어로 검증.

### 조건부 사용

#### `brainstorming`

권장 트리거:
- 요구사항이 불분명할 때.
- 유의미한 제품 또는 UX 선택이 필요할 때.
- 새로운 기능 정의 시.

다음의 경우 트리거하지 않음:
- 명확한 버그 수정.
- 문구(copy) 변경.
- 예상 동작이 알려진 고립된 표시 로직 수정.

#### `writing-plans`

권장 트리거:
- 3개 이상의 유의미한 구현 단계가 있을 때.
- 여러 상태 전이(state transitions)나 영속성 경계(persistence boundaries)가 있을 때.
- 사용자가 명시적으로 계획을 요청할 때.

다음의 경우 트리거하지 않음:
- 범위가 명확한 작은 버그 수정.

#### `executing-plans`

권장 트리거:
- 실제 계획이 존재하고, 작업이 구조화된 실행으로부터 이득을 얻을 만큼 충분히 클 때만.

다음의 경우 트리거하지 않음:
- 작은 직접 버그 수정.

#### `requesting-code-review`

권장 트리거:
- 5개 이상의 파일이 변경되었을 때.
- 영속성, 데이터 모델, 또는 동기화 로직을 건드렸을 때.
- 사용자가 리뷰를 요청할 때.
- 위험한 리팩토링 시.

다음의 경우 트리거하지 않음:
- 실제 통합 위험이 없는 한 작은 UI 레이블이나 포맷팅 수정 시.

### 자동 경로에서 제거

#### `finishing-a-development-branch`

권장 트리거:
- 사용자가 명시적으로 커밋, 머지, 푸시, 또는 PR 생성을 요청할 때만.

다음의 경우 트리거하지 않음:
- "구현 완료" 그 자체만으로는 실행하지 않음.

#### `code-reviewer` 또는 리뷰용 `general` 서브 에이전트

권장 트리거:
- 변경 범위나 사용자 요청에 의해 리뷰가 정당화될 때만.

다음의 경우 트리거하지 않음:
- 기본적으로 모든 인라인 실행에 대해 실행하지 않음.

## 리뷰 서브 에이전트를 위한 권장 트리거 규칙 (Recommended Trigger Rules For Review Subagents)

리뷰 서브 에이전트는 메인 에이전트보다 훨씬 더 엄격해야 합니다.

### 권장되는 허용 동작

- 코드 검사.
- diff 검사.
- 테스트 검사.
- 필요한 경우 집중적인 검증 실행.
- 결과(findings) 반환.

### 권장되는 금지 동작

- `using-superpowers` 호출 금지.
- `brainstorming` 호출 금지.
- `writing-plans` 호출 금지.
- `executing-plans` 호출 금지.
- `receiving-code-review` 호출 금지.
- `finishing-a-development-branch` 호출 금지.
- 메인 에이전트의 명시적 지시가 없는 한 새로운 서브 에이전트 파견 금지.

### 이유

이렇게 해야 리뷰 작업자가 순수하게 리뷰 작업자로 남을 수 있습니다.
워크플로우 재귀를 방지하고 토큰 및 지연 시간 비용을 줄일 수 있습니다.

## 권장되는 세 가지 개선된 흐름 (Three Recommended Improved Flows)

이 프로젝트 스타일에 맞춰 가장 추천하는 순서대로 나열했습니다.

### 1. 추천: 개인용 빠르고 안전한 인라인 (Personal Fast-Safe Inline)

흐름:

`using-superpowers -> 작업 분류 -> 버그인 경우 systematic-debugging -> test-driven-development -> 메인 에이전트의 인라인 구현 -> 집중 검증 -> 위험도가 높거나 요청 시 선택적 단일 리뷰`

언제 사용하는가:
- 명확한 버그 수정.
- 작은 UI 또는 상태 수정.
- 저위험 동작 변경.
- 개인 프로젝트 반복 작업.

현재 흐름과 비교하여 바뀌는 점:
- 명확한 버그 수정에 의무적인 브레인스토밍 없음.
- 의무적인 계획 문서 없음.
- 의무적인 리뷰 서브 에이전트 없음.
- 사용자 요청 전까지 브랜치 종료 워크플로우 없음.

장점:
- 가장 빠른 경로.
- 디버깅, TDD, 검증을 통해 정확성 유지.
- 워크플로우 오버헤드 최소화.
- `인라인`이라는 단어의 직관적 의미와 일치.

단점:
- 프로세스 문서화가 적음.
- 명시적인 리뷰 체크포인트가 적음.

선택 시 결과:
- 이번과 같은 버그 수정은 아마도 다음과 같이 진행되었을 것입니다:
  - `using-superpowers`
  - `systematic-debugging`
  - `test-driven-development`
  - 인라인 수정
  - 집중 테스트 검증
  - 완료

### 2. 균형형: 단일 리뷰 게이트를 포함한 개인용 인라인 (Personal Inline With One Review Gate)

흐름:

`using-superpowers -> 작업 분류 -> 모호성이 있을 경우 선택적인 가벼운 디자인 노트 -> test-driven-development -> 인라인 구현 -> 집중 검증 -> 변경 사항이 위험 임계치를 넘을 때만 한 번의 리뷰 서브 에이전트 파견 -> 완료`

언제 사용하는가:
- 중간 규모의 변경.
- 4~8개의 파일 수정.
- 상태 전이, 영속성, 또는 조정 로직 변경.
- 여전히 개인 프로젝트이지만 더 높은 안전성을 원할 때.

현재 흐름과 비교하여 바뀌는 점:
- 계획 수립은 가볍게 채팅 내에서 이루어질 수 있음.
- 리뷰 게이트는 한 번만.
- 첫 리뷰에서 중요한 것이 발견되지 않는 한 반복적인 재리뷰 루프 없음.
- 명시적인 통합 요청 없이는 브랜치 종료 스킬 없음.

장점:
- 속도와 안전의 좋은 균형.
- 통합 실수를 잡아냄.
- 현재의 전체 워크플로우보다 오버헤드가 적음.

단점:
- 순수하게 빠르고 안전한 인라인보다는 여전히 느림.
- 리뷰가 가치 있는 시점에 대한 명확한 기준이 필요함.

선택 시 결과:
- 이 버그 수정은 아마 디자인 문서나 계획 문서를 사용하지 않았을 것입니다.
- 구현 후 한 번의 리뷰 패스를 거쳤을 수 있지만, 여러 번의 루프는 없었을 것입니다.

### 3. 여기서는 비추천: 풀 팀 워크플로우 (Full Team Workflow)

흐름:

`using-superpowers -> systematic-debugging -> brainstorming -> writing-plans -> 실행 모드 선택 -> executing-plans -> 리뷰 서브 에이전트 -> receiving-code-review -> finishing-a-development-branch -> 최종 검증`

언제 사용하는가:
- 대규모 기능.
- 아키텍처 변경.
- 다인 협업 작업.
- 정식 리뷰를 통해 인계되거나 머지되어야 하는 작업.

현재 흐름과 비교하여 바뀌는 점:
- 본질적으로 이번 세션에서 일어난 일과 동일함.

장점:
- 가장 강력한 문서화 추적.
- 가장 강력한 프로세스 규율.
- 팀을 위한 명확한 체크포인트.

단점:
- 가장 높은 오버헤드.
- 느린 피드백 사이클.
- "문제 우선"이 아닌 "워크플로우 우선"으로 느껴지기 쉬움.
- 작은 개인 프로젝트 버그 수정의 기본값으로는 좋지 않음.

선택 시 결과:
- 이번 세션에서 일어났던 것과 똑같이 무거운 경로를 거치게 됨.

## 권장 선택안 (Recommended Choice)

이 저장소와 당신이 설명한 스타일을 고려할 때, 가장 좋은 기본값은 다음과 같습니다:

### `개인용 빠르고 안전한 인라인 (Personal Fast-Safe Inline)`

이것이 속도와 정확성 사이에서 최적의 트레이드오프를 제공합니다.

제안하는 기본 정책:

1. 항상 `using-superpowers`로 시작합니다.
2. 작업이 명확한 버그 수정인 경우 `systematic-debugging`을 사용합니다.
3. 수정이 동작을 변경하는 경우 `test-driven-development`를 사용합니다.
4. 메인 에이전트에서 인라인으로 구현합니다.
5. 집중 테스트로 검증합니다.
6. 위험도가 정당화될 때만 리뷰 서브 에이전트를 사용합니다.
7. 사용자가 명시적으로 통합 작업을 요청할 때만 브랜치 종료 워크플로우를 사용합니다.

## 구체적인 개선 조치 (Concrete Improvement Actions)

### 우선순위 1

`인라인 실행(Inline Execution)`의 의미를 다음과 같이 재정의:
- 자동적인 서브 에이전트 파견 없음.
- 사용자가 리뷰를 요청하거나 변경 사항이 정의된 위험 임계치를 넘지 않는 한 메인 에이전트 전용으로 수행.

### 우선순위 2

`brainstorming` 및 `writing-plans`에 작은 버그 수정에 대한 예외 사항 추가.

제안 규칙:
- 작업이 명확한 범위의 버그 수정이고, 기대 동작이 명시되어 있으며, 작은 국소 부위만 건드리는 경우 의무적인 디자인 및 계획 문서 워크플로우를 건너뜀.

### 우선순위 3

리뷰 서브 에이전트가 일반 워크플로우 스킬에 재진입하지 못하도록 제한.

제안 규칙:
- 리뷰 서브 에이전트는 검사 및 검증은 할 수 있으나 계획 수립, 리뷰 수신, 브랜치 종료, 또는 추가적인 에이전트 파견 흐름을 호출할 수 없음.

### 우선순위 4

`finishing-a-development-branch`를 선택 사항(opt-in)으로 변경.

제안 규칙:
- 사용자가 커밋, 머지, 푸시, 또는 PR 생성을 요청할 때만 실행.

### 우선순위 5

검증 명령어 범위 축소.

제안 규칙:
- 국소적인 버그 수정을 증명할 때는 저장소 전체나 환경 전체 테스트보다 프로젝트에 속한 집중적인 검증을 선호함.

## 최종 평가 (Final Assessment)

이번 세션에서 사실이었던 것들:

1. 구현 자체는 메인 에이전트에서 인라인으로 이루어졌습니다.
2. 현재의 `executing-plans -> requesting-code-review` 체인이 요구하기 때문에 리뷰 시에는 여전히 서브 에이전트가 사용되었습니다.
3. 리뷰 서브 에이전트가 내부적으로 스킬을 호출했다고 보고했습니다.
4. 서브 에이전트 결과가 돌아온 후 메인 에이전트가 더 많은 스킬을 트리거했습니다.
5. 이는 현재의 스킬 디자인하에서 프로세스적으로는 옳았습니다.
6. 하지만 빠르고 안전한 개인 프로젝트 워크플로우에는 최적의 맞춤이 아니었습니다.

가장 중요한 개선 사항은 다음 두 가지 의미를 명확히 분리하는 것입니다:

- `인라인 구현 (inline implementation)`
- `메인 에이전트 전용 실행 (main-agent-only execution)`

현재로서는 이 둘은 같은 것이 아닙니다.
